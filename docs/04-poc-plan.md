# QueueBus PoC 계획

## 1. PoC 목표

QueueBus PoC의 목표는 전국 단위 서비스 검증이 아니라, 혼잡한 광역버스 정류장 1곳과 노선 1~2개에서 위치 인증형 가상 대기열, AI 예측, 고정 대기번호와 실시간 내 앞 대기 인원 UX가 실제 운영 문제를 줄일 수 있는지 확인하는 것입니다.

핵심 검증 질문은 다음입니다.

- 정류장 반경 내 위치 인증으로 현장 도착자를 공정하게 대기열에 등록할 수 있는가?
- 예상 잔여좌석과 대기 순번을 결합해 사용자가 이해 가능한 탑승 가능성 정보를 제공할 수 있는가?
- 전체 대기자를 줄 세우지 않고 이번 차량 탑승 가능성이 높은 인원만 호출할 수 있는가?
- 고정 대기번호와 실시간 내 앞 대기 인원 안내가 "몇 번이세요?" 같은 구두 확인을 줄일 수 있는가?
- 운영자가 정류장별·노선별·시간대별 혼잡 데이터를 의사결정에 활용할 수 있는가?

## 2. 대상 정류장 선정 기준

PoC 대상 정류장은 다음 기준으로 선정합니다.

- 퇴근 시간대 물리적 대기줄이 반복적으로 발생하는 광역버스 정류장
- 노선별 대기줄이 혼재해 순번 갈등 가능성이 있는 정류장
- 보행로 점유 또는 주변 보행자 불편이 관찰되는 정류장
- 버스 도착 정보와 잔여좌석 정보를 공식 API 또는 수기 관찰로 확보할 수 있는 정류장
- 기존 바닥 버스번호 위치와 앱의 호출 대상 대기번호 구간을 연결해 현장 UX 설명이 가능한 정류장
- 지자체/운수사 협업 가능성이 있는 정류장

초기 문서에서는 특정 정류장명을 확정하지 않고 `혼잡 광역버스 정류장 A`로 가정합니다. 제출 전 실제 후보 정류장 1곳을 정해 현장 사진, 관찰표, 주변 대기 공간 정보를 보완합니다.

## 3. 대상 노선 선정 기준

- 출퇴근 시간대 대기 수요가 높고 만차 또는 잔여좌석 부족이 발생하는 노선
- 배차 간격이 너무 짧지 않아 대기 불확실성이 체감되는 노선
- 정류장 도착 전 남은 정류장 수와 잔여좌석 변화가 관찰 가능한 노선
- PoC 기간에 현장 관찰자가 탑승 인원과 미탑승 인원을 확인할 수 있는 노선

초기 PoC는 1개 정류장, 1개 핵심 노선으로 시작하고, 대시보드에서는 비교를 위해 2개 노선 mock 데이터를 함께 표시합니다.

## 4. 현장 관찰 방법

현장 관찰은 퇴근 피크 시간대 1~2시간을 기준으로 수행합니다.

관찰자는 버스 도착 직전 대기 인원, 차량 도착 시간, 탑승 인원, 미탑승 인원, 보행로 점유 여부, 예상 줄 길이, 날씨, 온도, 특이사항을 기록합니다.

기록 항목은 `data/field-observation-template.csv`를 사용합니다.

- observation_date
- time
- station_name
- route_name
- waiting_count_before_bus_arrival
- bus_arrival_time
- boarded_count
- left_waiting_count
- estimated_queue_length_meters
- sidewalk_blocked_yn
- weather
- temperature
- memo

현장 관찰 시 개인 얼굴, 차량 번호판, 개별 위치 경로 등 불필요한 개인정보는 수집하지 않습니다.

## 5. 데이터 수집 방법

### MVP 단계

- 버스 잔여좌석/ETA: mock CSV 사용
- 체크인/호출/탑승 이벤트: sample event CSV 사용
- 날씨/기온: 수기 입력 또는 mock 값 사용
- 위치 인증: 사용자 좌표와 정류장 좌표의 Haversine 거리 계산

### PoC 확장 단계

- 공식 버스정보 API 연동 가능성 검토
- 정류장 좌표 DB 구축
- 날씨 API 연동
- 현장 관찰 데이터와 앱 이벤트 데이터 결합
- 탑승 성공/실패 결과를 예측 모델 학습 데이터로 전환

네이버지도나 카카오맵 화면을 크롤링하지 않습니다. 지도 화면 크롤링 대신 공식 API, 공개 데이터, 수기 관찰, 사용자 동의 기반 앱 이벤트 데이터를 사용합니다.

## 6. 검증 지표

| 구분 | 지표 | 측정 방법 |
| --- | --- | --- |
| 위치 인증 | 체크인 가능 판정 정확도 | 정류장 반경 내/외 테스트 좌표 비교 |
| 사용자 흐름 | 체크인 완료율 | 정류장 선택 후 대기번호 발급 완료 비율 |
| 예측 | 탑승 가능성 예측 정확도 | 예측 가능성 구간과 실제 탑승 결과 비교 |
| 호출 | 호출 응답률 | 호출된 사용자 중 기존 노선 대기 위치 도착 처리 비율 |
| 호출 | 노쇼율 | 호출 후 미도착/미응답 비율 |
| 혼잡 완화 | 평균 물리적 줄서기 시간 | 기존 줄서기 추정 시간과 호출 후 정렬 시간 비교 |
| 운영 | 대기 수요 데이터 완성도 | 정류장/노선/시간대별 이벤트 누락률 |
| 사회적 가치 | 보행로 점유 완화 | 현장 관찰자의 sidewalk_blocked_yn 변화 |
| 만족도 | 사용자 이해도/만족도 | 간단 설문 또는 인터뷰 |

## 7. PoC 일정

| 단계 | 기간 | 내용 | 산출물 |
| --- | --- | --- | --- |
| 준비 | 1주 | 후보 정류장/노선 선정, 관찰표 확정 | PoC 대상 정의 |
| 관찰 | 1주 | 퇴근 시간대 현장 관찰 1회 이상 | 관찰 CSV, 현장 메모 |
| 프로토타입 | 1~2주 | 사용자 화면, 운영자 대시보드, 예측 로직 구현 | 웹 PoC |
| 검증 | 1주 | mock 데이터와 현장 관찰 결과 비교 | 예측 오차 요약 |
| 보완 | 1주 | 사업계획서, 발표자료, 개인정보 보호 설명 보완 | 제출 패키지 |

## 8. PoC 결과 리포트 구성

- 대상 정류장/노선 개요
- 관찰 일시와 날씨 조건
- 차량별 대기 인원, 탑승 인원, 미탑승 인원
- QueueBus 적용 시 예상 호출 인원과 호출 대상 대기번호 구간
- 실제 결과와 예측 결과 차이
- 사용자/운영자 관점 기대효과
- 실제 시범 운영을 위한 보완 과제
