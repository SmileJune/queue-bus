# QueueBus AI 모델 상세 설계

작성일: 2026-06-02  
목적: QueueBus가 사용할 AI의 종류, 예측 기술, 입력 데이터, 출력 결과, MVP 구현 방식, 고도화 방식을 공모전 제출·발표용으로 정리합니다.

## 1. 전체 구조

QueueBus의 AI는 각각 별도 SaaS 서비스가 아닙니다. QueueBus라는 하나의 정류장 대기 관리 SaaS 안에 들어가는 내부 예측 모듈입니다. 문서에서 `SeatFlow AI`, `Demand Forecast AI`처럼 이름을 붙인 이유는 심사위원이 기능을 구분해 이해하기 쉽게 하기 위한 것이며, 각각을 별도 제품으로 판매한다는 의미는 아닙니다.

QueueBus의 AI는 하나의 거대한 모델이 아니라, 정류장 운영 문제를 여러 예측 문제로 나눈 조합형 AI 모듈 구조입니다.

핵심 원칙:

- 대기 순번은 AI가 정하지 않습니다. 순번은 위치 인증 완료 시점 기준 선착순입니다.
- AI는 “누가 먼저 탈지”가 아니라 “몇 명이 탈 수 있는지”, “내 순번이면 이번 차를 탈 수 있는지”, “몇 번까지 호출할지”, “어느 정류장이 위험한지”를 예측합니다.
- 초기 MVP부터 모든 기능을 단순 규칙으로만 두지 않습니다. 현재 GBIS 수집 데이터가 이미 있으므로, 핵심 기능인 SeatFlow AI는 경량 학습 모델로 시작하고 나머지 모듈은 규칙 기반 보조 로직으로 시작합니다.
- PoC에서 사용자 체크인·호출·탑승 결과 데이터가 쌓이면 Boarding Probability, Call Optimizer, Left-behind Risk까지 학습형 모델로 확장합니다.

전체 AI 모듈 구성:

| AI 이름 | 예측 대상 | 주요 사용자 | MVP 기술 | 고도화 기술 |
| --- | --- | --- | --- | --- |
| SeatFlow AI | 정류장 도착 시 예상 잔여좌석·탑승 가능 인원 | 이용자, 운영자 | LightGBM/XGBoost 경량 회귀 + 규칙 fallback | 추가 실사·체크인 데이터 반영 온라인 재학습 |
| Demand Forecast AI | 정류장·시간대별 예상 대기 인원 | 운영자 | 시간대 평균, 최근 추세 | 시계열 모델, 회귀 모델 |
| Boarding Probability AI | 사용자별 이번 차/다음 차 탑승 가능성 | 이용자 | 순번-좌석 비교, 노쇼율 보정 | 로지스틱 회귀, 확률 보정 |
| Call Optimizer AI | 호출 인원·호출 시점·대기번호 구간 | 이용자, 운영자 | 규칙 기반 최적화 | 강화학습 전 단계의 정책 최적화 |
| Left-behind Risk AI | 미탑승 발생·반복 위험 | 운영자 | 임계값 기반 감지 | 분류 모델, 이벤트 예측 |
| Congestion & Safety AI | 보행로 혼잡·날씨 위험 | 운영자 | 가중 점수 | 위험도 분류 모델 |
| Anomaly Detection AI | 위치조작·반복 노쇼·비정상 체크인 | 운영자 | 규칙 기반 | Isolation Forest, One-Class SVM |
| Operator Insight AI | 운영자용 요약·권고 문장 | 운영자 | 템플릿 생성 | LLM 기반 리포트 요약 |

### 1-1. AI를 전혀 모르는 사람을 위한 기술 설명

이 문서에서 말하는 `MVP 기술`은 “초기 제품에서 바로 구현할 수 있고, 심사위원에게 작동 원리를 설명할 수 있는 예측 방식”입니다. QueueBus는 이미 GBIS 잔여좌석·차량 위치 데이터를 수집하고 있으므로, 핵심 기능인 SeatFlow AI는 처음부터 경량 학습 모델로 시작하는 것이 더 설득력 있습니다. 다만 사용자 체크인 데이터가 아직 없는 기능은 규칙, 평균, 비율, 점수 계산으로 보조합니다.

이 문서에서 말하는 `고도화 기술`은 “데이터가 쌓인 뒤 컴퓨터가 과거 패턴을 학습해 더 정확히 예측하는 방식”입니다. 예를 들어 지금은 `잔여좌석 23석 - 중간 정류장 예상 탑승 14명 = 9석`처럼 직접 정한 공식으로 계산하지만, 고도화 단계에서는 과거 수천 건의 차량 이동·좌석 변화·대기열 결과를 모델이 학습해 “이 시간대 이 정류장에서는 실제로 몇 석이 남는 경우가 많았는지”를 자동으로 찾아냅니다.

핵심 용어:

| 용어 | 쉬운 설명 | QueueBus 예시 |
| --- | --- | --- |
| 규칙 기반 | 사람이 정한 공식으로 계산하는 방식 | 잔여좌석에서 중간 정류장 탑승 예상 인원을 빼기 |
| 이동평균 | 최근 여러 값의 평균으로 다음 값을 추정하는 방식 | 최근 5대의 이주택지 대기 인원 평균을 다음 대기 인원 예측에 사용 |
| 회귀 모델 | 숫자를 예측하는 AI | 다음 차량이 이주택지에 도착할 때 잔여좌석이 몇 석인지 예측 |
| 분류 모델 | 여러 상태 중 하나를 고르는 AI | 이번 차량에서 미탑승이 발생할지 `예/아니오`로 예측 |
| 로지스틱 회귀 | 결과가 `성공/실패`일 때 확률을 예측하는 기본 모델 | 내 순번이 이번 차량에 탈 확률이 72%인지 계산 |
| LightGBM/XGBoost | 표 형태 데이터에서 성능이 좋은 예측 모델 | 정류장, 시간대, 잔여좌석, 배차간격을 보고 잔여좌석을 예측 |
| 시계열 모델 | 시간 흐름의 반복 패턴을 학습하는 모델 | 평일 08:30마다 대기 인원이 증가하는 패턴 학습 |
| 이상탐지 | 평소와 다른 행동을 찾아내는 모델 | 100m 밖에서 체크인하거나 반복 노쇼하는 패턴 감지 |
| LLM | 자연어 문장을 생성·요약하는 AI | 운영자에게 “이 정류장은 다음차 안내가 필요합니다” 문장 생성 |

QueueBus의 기술 단계는 다음처럼 이해하면 됩니다.

| 단계 | 기술 수준 | 무엇을 보여주는가 | 예 |
| --- | --- | --- | --- |
| MVP | 경량 학습 모델 + 규칙 fallback | 실제 수집 데이터로 학습한 AI 예측이 이미 작동함 | GBIS 좌석 변화 데이터를 학습해 이주택지 도착 시 잔여좌석 예측 |
| PoC | 회귀·분류 모델 확장 | 사용자 체크인·호출·탑승 결과로 예측 정확도를 개선 | 내 대기번호 탑승 성공 확률, 미탑승 발생 여부 예측 |
| 사업화 | 시계열·이상탐지·LLM | 정류장 운영 자동화와 리포트 고도화 | 반복 혼잡 정류장 알림, 운영 리포트 자동 작성 |

### 1-2. 왜 처음부터 일부 고도화 모델로 시작하는가

QueueBus가 의미 있는 AI 서비스로 보이려면 단순 공식만으로는 부족합니다. 특히 심사위원은 “AI라고 하지만 결국 엑셀 계산 아닌가?”라고 볼 수 있습니다. 따라서 초기 MVP의 중심은 `SeatFlow AI 학습 모델`로 잡는 것이 좋습니다.

가능한 이유:

- 이미 GBIS 스냅샷이 5만 행 이상 쌓여 있습니다.
- `gbis-boarded-estimates.csv`에 차량별 좌석 변화 기반 탑승 추정치가 있습니다.
- 정류장별, 차량별, 시간대별 잔여좌석 패턴을 학습할 수 있습니다.
- 예측 대상이 이미지·음성처럼 복잡하지 않고 표 형태 숫자 데이터입니다.
- LightGBM/XGBoost는 작은 팀도 적용 가능한 성숙한 오픈소스 모델입니다.

초기부터 학습형으로 시작할 모델과 아닌 모델을 구분합니다.

| 구분 | 초기 전략 | 이유 |
| --- | --- | --- |
| SeatFlow AI | 처음부터 LightGBM/XGBoost 또는 Random Forest 회귀 모델 적용 | GBIS 좌석·위치 데이터가 이미 충분히 있음 |
| Demand Forecast AI | 초기에는 시간대 평균, 이후 체크인 데이터가 쌓이면 학습 모델 적용 | 실제 대기 인원 데이터는 아직 부족함 |
| Boarding Probability AI | 초기에는 규칙 기반, 이후 사용자 탑승 성공 데이터로 분류 모델 적용 | 개인별 탑승 성공 라벨이 아직 없음 |
| Call Optimizer AI | 초기에는 규칙 기반, 이후 호출 결과로 정책 최적화 | 호출 응답·노쇼 데이터가 아직 없음 |
| Left-behind Risk AI | 초기에는 SeatFlow 결과 기반 위험 분류, 이후 실사·체크인 데이터로 분류 모델 적용 | 미탑승 라벨이 적지만 핵심 운영 지표임 |

즉, 처음부터 모든 AI를 고도화 모델로 만들겠다는 전략은 위험합니다. 대신 현재 데이터가 충분한 SeatFlow AI는 학습형 모델로 시작하고, 아직 데이터가 없는 대기열·호출 관련 모델은 규칙 기반으로 시작한 뒤 빠르게 학습형으로 전환하는 전략이 가장 현실적입니다.

## 2. 실시간 데이터 흐름

현재 미니PC는 GBIS 데이터를 수집하고 있으며, 대시보드 API로 최근 상태를 제공합니다.

현재 확인된 실시간 데이터 예:

| 항목 | 값 |
| --- | --- |
| API | `http://127.0.0.1:4175/api/summary?windowHours=2` |
| 최신 수집 지연 | 약 25초 |
| 누적 스냅샷 | 54,051행 |
| 최근 2시간 차량 샘플 | 1,053건 |
| 탑승 추정 행 | 1,830건 |
| 대상 정류장 | 14곳 |

정류장별 예측에 쓰는 주요 필드:

| 필드 | 의미 | 예 |
| --- | --- | --- |
| `targetStationName` | 예측 대상 정류장 | 이주택지.상록.경남아파트 |
| `targetStationSeq` | 노선 내 정류장 순번 | 9 |
| `mobileNo` | 정류장 번호 | 55305 |
| `avgSeats` | 최근 창 평균 잔여좌석 | 9.6 |
| `minSeats` | 최근 창 최소 잔여좌석 | 0 |
| `nextVehicles` | 접근 중인 다음 차량 목록 | 경기76아4810, 잔여 23석 |
| `remainingStops` | 목표 정류장까지 남은 정류장 수 | 4 |
| `estimateRows` | 좌석 변화 기반 탑승 추정 | 1,830행 |

예시:

```json
{
  "targetStationName": "이주택지.상록.경남아파트",
  "mobileNo": "55305",
  "avgSeats": 9.6,
  "minSeats": 0,
  "nextVehicles": [
    {
      "plateNo": "경기76아4810",
      "currentStationSeq": 5,
      "remainSeatCount": 23,
      "remainingStops": 4
    }
  ]
}
```

이 데이터만으로도 정류장별 `다음 차량 승차 가능성`, `저좌석 위험`, `다음차 안내 필요 여부`를 자동 갱신할 수 있습니다. 사용자 체크인 데이터가 붙으면 `내 순번 탑승 가능성`과 `호출 대기번호 구간`까지 계산할 수 있습니다.

## 3. SeatFlow AI

### 3-1. 목적

SeatFlow AI는 특정 차량이 목표 정류장에 도착했을 때 남아 있을 좌석 수와 실제 탑승 가능 인원을 예측합니다.

사용 예:

- “이주택지 정류장에 다음 M4137이 도착할 때 몇 명 정도 탈 수 있는가?”
- “지금 잔여좌석은 23석이지만, 중간 정류장을 지나면 몇 석으로 줄어들 가능성이 있는가?”
- “0석 가능성이 높으니 다음차 안내를 해야 하는가?”

### 3-2. 입력 데이터

| 입력 | 설명 |
| --- | --- |
| 현재 차량 잔여좌석 | GBIS `remainSeatCount` |
| 현재 차량 정류장 순번 | GBIS `currentStationSeq` |
| 목표 정류장 순번 | 예: 이주택지 순번 9 |
| 남은 정류장 수 | 목표 순번 - 현재 순번 |
| 중간 정류장별 평균 탑승 수 | `gbis-boarded-estimates.csv`에서 계산 |
| 중간 정류장별 평균 하차 수 | 초기에는 0 또는 보수값, 실사/PoC 후 보정 |
| 요일·시간대 | 평일 출근, 퇴근, 휴일 외출 등 |
| 최근 저좌석 이력 | 최근 2시간 `minSeats`, `avgSeats`, 0석 비율 |

### 3-3. MVP 예측 방식

SeatFlow AI는 QueueBus의 핵심이므로 MVP부터 경량 학습 모델로 시작합니다. 규칙 기반 계산은 모델이 아직 학습하지 못한 정류장이나 데이터가 부족한 시간대의 fallback으로만 사용합니다.

쉽게 말하면, SeatFlow AI의 MVP는 “과거 차량들이 비슷한 상황에서 목표 정류장에 몇 석을 남기고 도착했는지”를 학습하는 모델입니다. 예를 들어 `현재 5번 정류장`, `잔여좌석 23석`, `목표는 9번 이주택지`, `평일 08:45`, `최근 이 정류장 0석 이력 있음` 같은 조건을 입력하면, 과거 데이터를 바탕으로 목표 정류장 도착 시 잔여좌석을 예측합니다.

MVP 학습 모델:

| 항목 | 내용 |
| --- | --- |
| 모델 후보 | LightGBM 회귀, XGBoost 회귀, Random Forest 회귀 |
| 예측값 | 목표 정류장 도착 시 예상 잔여좌석 또는 탑승 가능 인원 |
| 학습 데이터 | `gbis-seat-snapshots.csv`, `gbis-boarded-estimates.csv` |
| 입력 특징 | 현재 잔여좌석, 현재 정류장 순번, 목표 정류장 순번, 남은 정류장 수, 시간대, 요일, 최근 평균 좌석, 최근 0석 비율 |
| 출력 예 | “이주택지 도착 예상 잔여좌석 10석, 저좌석 위험 보통” |

학습용 데이터 예:

| current_station_seq | target_station_seq | remaining_stops | current_seats | time_bucket | recent_avg_seats | actual_target_seats |
| ---: | ---: | ---: | ---: | --- | ---: | ---: |
| 5 | 9 | 4 | 23 | 08:45 | 9.6 | 10 |
| 7 | 9 | 2 | 2 | 08:45 | 9.6 | 2 |
| 5 | 10 | 5 | 23 | 08:45 | 8.5 | 6 |

이 표에서 마지막 열 `actual_target_seats`가 모델이 맞혀야 하는 정답입니다. 모델은 앞의 조건들을 보고 정답에 가까운 잔여좌석을 예측하도록 학습합니다.

규칙 기반 fallback은 다음과 같습니다.

```text
expectedRemainingSeats =
  currentRemainingSeats
  - expectedBoardingBeforeTarget
  + expectedAlightingBeforeTarget

expectedBoardingBeforeTarget =
  sum(중간 정류장별 평균 탑승 수)

expectedAlightingBeforeTarget =
  sum(중간 정류장별 평균 하차 수)
```

예시:

```text
다음 차량: 경기76아4810
현재 위치: 5번 정류장
목표 정류장: 9번 이주택지
현재 잔여좌석: 23석
남은 정류장: 4개
중간 정류장 예상 탑승: 14명
중간 정류장 예상 하차: 1명

예상 잔여좌석 = 23 - 14 + 1 = 10석
```

따라서 QueueBus는 이주택지에서 약 10명까지 탑승 가능하다고 예측하고, 대기열이 10명 이하이면 일반 호출, 10명을 초과하면 일부는 다음차 안내를 제공합니다.

### 3-4. 실사 데이터 보정 예시

2026-06-02 이주택지 실사:

```text
차량 1040
도착 전 빈자리: 2석
이주택지 하차: 1명
실제 탑승: 3명
미탑승: 4명
```

잔여좌석만 쓰면 2명 탑승으로 예측하지만, 하차 1명을 반영하면 3명 탑승으로 실제값과 맞습니다.

```text
seatOnlyPrediction = 2명
alightingAdjustedPrediction = 2 + 1 = 3명
actualBoarded = 3명
error = 0명
```

이 사례는 SeatFlow AI가 “잔여좌석 + 하차 추정”을 함께 써야 하는 이유를 보여줍니다.

### 3-5. 고도화 모델

MVP부터 학습 모델을 적용하되, 고도화 단계에서는 모델의 입력과 정답 데이터를 더 정교하게 만듭니다.

고도화 단계에서는 사람이 직접 “정류장마다 평균 몇 명 탄다”를 정하지 않습니다. 모델이 과거 데이터를 보고 패턴을 학습합니다. 예를 들어 `평일 08:45`, `이주택지 전 4개 정류장`, `현재 잔여좌석 23석`, `직전 차량 미탑승 발생`, `배차간격 12분` 같은 조건에서 실제로 몇 석이 남았는지를 수천 건 학습합니다. 그다음 비슷한 상황이 오면 목표 정류장 도착 시 잔여좌석을 자동으로 예측합니다.

`LightGBM`과 `XGBoost`는 표 형태 데이터에 강한 회귀 모델입니다. QueueBus 데이터는 이미지나 음성이 아니라 정류장 번호, 시간대, 잔여좌석, 배차간격, 대기 인원 같은 표 데이터이므로 이 모델들이 현실적인 선택입니다.

예측 대상:

```text
y = 목표 정류장 도착 시 잔여좌석
또는
y = 목표 정류장에서 실제 탑승 가능 인원
```

특징값:

| 특징값 | 예 |
| --- | --- |
| current_remaining_seats | 23 |
| remaining_stops | 4 |
| target_station_seq | 9 |
| day_type | weekday |
| time_bucket | 08:45 |
| route_id | M4137 |
| station_recent_avg_boarding | 3.2 |
| station_recent_zero_seat_rate | 0.42 |
| headway_minutes | 12 |

고도화 방향:

| 단계 | 설명 |
| --- | --- |
| 1차 | GBIS 좌석 스냅샷만으로 목표 정류장 잔여좌석 예측 |
| 2차 | 좌석 변화 기반 탑승 추정치를 추가해 중간 정류장 승차 수요 반영 |
| 3차 | 현장 실사와 앱 체크인 데이터를 추가해 실제 미탑승과 하차 보정 |
| 4차 | 모델 예측 오차를 매일 자동 측정하고 주기적으로 재학습 |

평가 지표:

- 잔여좌석 MAE
- 실제 탑승 가능 인원 MAE
- 0석/저좌석 예측 정확도
- 과소 호출/과다 호출 비율

## 4. Demand Forecast AI

### 4-1. 목적

정류장·노선·시간대별로 앞으로 몇 명이 대기할지 예측합니다.

사용 예:

- “평일 08:30 이주택지 서울방향에는 몇 명 정도 대기할 가능성이 높은가?”
- “퇴근 17:30 명동입구는 다음차 안내가 반복적으로 필요한가?”
- “어느 정류장에 현장 안내 인력을 먼저 배치해야 하는가?”

### 4-2. 입력 데이터

| 입력 | 설명 |
| --- | --- |
| 사용자 체크인 수 | 정류장 반경 내 체크인 이벤트 |
| 시간대 | 15분 단위 bucket |
| 요일/공휴일 | 평일, 휴일 |
| 최근 미탑승 인원 | 앞 차량에서 못 탄 사람 |
| 배차 간격 | 차량 간 headway |
| 날씨 | 폭염, 우천, 한파 |
| 과거 같은 시간대 평균 | 최근 N일 평균 |

### 4-3. MVP 예측 방식

초기에는 이동평균과 최근 변화량을 씁니다.

Demand Forecast AI의 MVP는 “이 시간대에는 보통 몇 명이 기다렸는지”를 기준으로 다음 대기 인원을 추정합니다. 예를 들어 최근 며칠 동안 평일 08:30 이주택지에서 평균 7명이 기다렸고, 방금 전 차량에서 4명이 못 탔다면 다음 차량 전 대기 인원은 평균 7명에 이월 수요 4명을 더해 계산합니다.

```text
expectedWaitingCount =
  timeBucketAverage
  + recentCheckInDelta
  + failedBoardingCarryOver
```

예시:

```text
08:30 이주택지 평균 대기: 7명
최근 5분 신규 체크인 증가: +2명
직전 차량 미탑승 이월: +4명

예상 대기 인원 = 7 + 2 + 4 = 13명
```

이 값이 SeatFlow AI의 예상 탑승 가능 인원 10명보다 크면 3명은 다음차 안내 대상입니다.

### 4-4. 고도화 모델

PoC 이후에는 시계열 회귀 모델을 쓸 수 있습니다.

고도화 단계에서는 정류장별 반복 패턴을 모델이 학습합니다. 출근 시간에는 07:40부터 대기 인원이 늘고, 08:30에 피크가 오고, 09:20 이후 줄어드는 식의 시간 흐름이 있습니다. 시계열 모델은 이런 “시간에 따른 반복 패턴”을 예측하는 데 쓰입니다. 회귀 모델은 여기에 날씨, 공휴일, 배차간격, 직전 미탑승 같은 변수까지 함께 반영해 숫자를 예측합니다.

후보:

- LightGBM 회귀: 표 형태 데이터에 강하고 설명이 쉬움
- XGBoost 회귀: 작은 데이터에서도 안정적
- Prophet/ARIMA: 정류장별 반복 패턴이 뚜렷할 때 적합
- Temporal Fusion Transformer: 장기적으로 데이터가 많아졌을 때 사용

평가 지표:

- 대기 인원 MAE
- 피크 시간대 과소예측률
- 혼잡 등급 정확도

## 5. Boarding Probability AI

### 5-1. 목적

개별 사용자에게 이번 차량 또는 다음 차량을 탈 가능성을 안내합니다.

사용 예:

- “내 대기번호가 12번인데 이번 차를 탈 수 있나?”
- “이번 차 가능성이 낮으면 계속 줄 서지 말고 다음차를 기다려도 되나?”

### 5-2. 입력 데이터

| 입력 | 설명 |
| --- | --- |
| 내 대기번호 | 위치 인증 후 발급 |
| 내 앞 대기 인원 | 탑승, 취소, 노쇼 반영 |
| 예상 탑승 가능 인원 | SeatFlow AI 결과 |
| 노쇼율 | 호출 후 미도착 비율 |
| 버스 ETA | 도착 예정 시간 |
| 다음 차량 예상 | 다음 차량 SeatFlow 결과 |

### 5-3. MVP 예측 방식

Boarding Probability AI의 MVP는 “내 순번이 예상 탑승 가능 인원 안에 들어오는가”를 보는 방식입니다. 여기에 노쇼율을 반영합니다. 예를 들어 내 앞에 8명이 있어도 그중 20%가 호출 후 오지 않는다면 실제로는 6~7명 앞에 있는 것처럼 계산합니다.

```text
expectedNoShowCount = usersAhead * noShowRate
effectivePosition = queuePosition - expectedNoShowCount
margin = expectedBoardableCount - effectivePosition
```

확률화 예시:

```text
if margin >= 3: 탑승 가능성 높음, 80~95%
if -2 <= margin < 3: 탑승 가능성 보통, 35~75%
if margin < -2: 탑승 가능성 낮음, 5~35%
```

예시:

```text
내 대기번호: 9번
내 앞 대기 인원: 8명
노쇼율: 20%
예상 노쇼: 8 * 0.2 = 1.6명
유효 순번: 9 - 1.6 = 7.4번
예상 탑승 가능 인원: 10명

margin = 10 - 7.4 = 2.6
이번 차량 탑승 가능성: 보통~높음
```

### 5-4. 고도화 모델

로지스틱 회귀 또는 Gradient Boosting 분류 모델을 쓸 수 있습니다.

고도화 단계에서는 “이번 차를 탔는가/못 탔는가”라는 실제 결과를 학습합니다. 로지스틱 회귀는 어떤 조건일 때 탑승 성공 확률이 올라가거나 내려가는지 설명하기 쉬운 모델입니다. 예를 들어 내 앞 대기 인원이 많을수록 확률은 내려가고, 예상 잔여좌석이 많을수록 확률은 올라갑니다. Gradient Boosting 모델은 이런 관계가 단순 직선이 아닐 때 더 정교하게 예측합니다.

예측 대상:

```text
y = 이번 차량 탑승 성공 여부
```

특징값:

| 특징값 | 예 |
| --- | --- |
| queue_position | 9 |
| users_ahead | 8 |
| expected_boardable_count | 10 |
| no_show_rate_recent | 0.18 |
| eta_minutes | 4 |
| route_headway_minutes | 12 |
| station_risk_level | high |

평가 지표:

- Brier Score: 확률 예측 품질
- Calibration Error: 70%라고 안내한 그룹이 실제 70% 탑승했는지
- AUC: 탑승/미탑승 구분 성능

## 6. Call Optimizer AI

### 6-1. 목적

이번 차량에 몇 명을 언제 호출할지 결정합니다.

사용 예:

- “탑승 가능 인원이 10명일 때 몇 번까지 호출할까?”
- “노쇼가 있으니 10명보다 더 많이 부를까?”
- “버스 도착 몇 분 전에 호출해야 줄이 짧게 유지될까?”

### 6-2. MVP 예측 방식

Call Optimizer AI의 MVP는 “예상 탑승 가능 인원보다 약간 더 많이 부르는 규칙”입니다. 호출한 사람이 모두 오지는 않기 때문에 노쇼율을 반영합니다. 단, 너무 많이 부르면 다시 정류장 앞에 줄이 몰리므로 최대 호출 인원을 둡니다.

```text
callCount =
  min(maxCallCount, ceil(expectedBoardableCount / (1 - noShowRate)))

callTimeBeforeArrival =
  averageWalkToQueueSeconds + alignmentBufferSeconds
```

예시:

```text
예상 탑승 가능 인원: 10명
최근 노쇼율: 20%
최대 호출 인원: 14명

호출 인원 = ceil(10 / 0.8) = 13명
호출 대상 = 대기번호 1~13번
호출 시점 = 도착 120초 전
```

혼잡할 때는 보수적으로 조정합니다.

```text
if stationRiskLevel == "위험":
  callCount = min(callCount, expectedBoardableCount + 1)
```

### 6-3. 고도화 모델

초기에는 규칙 기반 최적화가 적합합니다. 이후에는 “호출 정책”을 학습할 수 있습니다.

고도화 단계에서는 여러 호출 전략을 비교합니다. 예를 들어 10명 탈 수 있을 때 10명을 부르면 노쇼 때문에 빈자리가 생길 수 있고, 15명을 부르면 정류장 앞이 다시 혼잡해질 수 있습니다. 모델은 과거 호출 결과를 보고 “이 정류장, 이 시간대, 이 날씨에서는 몇 명을 부르는 것이 가장 손실이 적었는지”를 학습합니다.

학습 대상:

- 너무 적게 호출하면 빈 좌석이 남음
- 너무 많이 호출하면 다시 줄 혼잡이 생김
- 적절한 호출은 탑승률을 높이고 물리적 줄서기를 줄임

목적 함수 예:

```text
loss =
  emptySeatPenalty * 남은 빈자리
  + crowdingPenalty * 과다 호출 인원
  + leftBehindPenalty * 미탑승 인원
  + waitingPenalty * 물리적 줄서기 시간
```

PoC 단계에서는 강화학습까지 갈 필요 없이, 여러 호출 정책을 시뮬레이션하고 손실이 가장 작은 규칙을 선택하면 충분합니다.

## 7. Left-behind Risk AI

### 7-1. 목적

실제 또는 예상 미탑승 위험을 감지합니다.

운영기관 관점에서 가장 중요한 AI입니다. 이용자에게는 불편이고, 운수사·지자체에게는 배차 검토와 현장 관리가 필요한 신호입니다.

### 7-2. 입력 데이터

| 입력 | 설명 |
| --- | --- |
| 예상 대기 인원 | Demand Forecast 결과 |
| 예상 탑승 가능 인원 | SeatFlow 결과 |
| 실제 잔여 대기 | 앱 대기열에서 남은 사람 |
| 좌석 0석 여부 | GBIS 잔여좌석 |
| 후속 차량 해소 여부 | 다음 차량에서도 남았는지 |

### 7-3. MVP 예측 방식

Left-behind Risk AI의 MVP는 예상 대기 인원과 예상 탑승 가능 인원을 비교합니다. 대기 인원이 더 많으면 미탑승이 생긴다고 봅니다. 또한 실제 관측이나 앱 이벤트에서 미탑승이 연속으로 발생하면 위험 등급을 올립니다.

```text
expectedLeftBehind =
  max(expectedWaitingCount - expectedBoardableCount, 0)

leftBehindRisk =
  if expectedLeftBehind >= 5: 위험
  else if expectedLeftBehind >= 1: 높음
  else: 보통
```

실사 예:

```text
1040 차량
대기 7명
탑승 3명
미탑승 4명

1043 차량
대기 5명
탑승 1명
남은 대기 4명

판정: 미탑승 반복, 위험
```

### 7-4. 고도화 모델

분류 모델로 바꿀 수 있습니다.

고도화 단계에서는 “다음 차량에서 미탑승이 발생할지”를 `예/아니오`로 예측합니다. 분류 모델은 과거에 미탑승이 발생했던 상황과 발생하지 않았던 상황을 비교해 학습합니다. 예를 들어 평균 잔여좌석이 낮고, 직전 차량 미탑승이 있었고, 배차간격이 길면 미탑승 위험을 높게 판단합니다.

예측 대상:

```text
y = 다음 차량 도착 후 미탑승 발생 여부
y = 미탑승 인원이 3명 이상인지 여부
```

모델 후보:

- 로지스틱 회귀: 설명이 쉬움
- Random Forest: 비선형 패턴에 강함
- LightGBM: 표 데이터에 강하고 성능 좋음

평가 지표:

- Precision: 위험 알림이 실제 위험이었는지
- Recall: 실제 미탑승을 놓치지 않았는지
- F1-score: 정밀도와 재현율 균형

## 8. Congestion & Safety AI

### 8-1. 목적

줄 길이, 대기시간, 날씨 위험을 합쳐 안전·혼잡 위험을 판단합니다.

### 8-2. MVP 예측 방식

Congestion & Safety AI의 MVP는 여러 위험 요소를 점수로 합산합니다. 대기 인원이 많을수록, 예상 대기시간이 길수록, 날씨가 나쁠수록 위험 점수가 올라갑니다. 이 점수는 운영자 대시보드에서 `낮음/보통/높음/위험`으로 표시됩니다.

```text
congestionScore =
  waitingCount * 0.5
  + expectedWaitMinutes * 0.3
  + weatherRiskScore * 0.2
```

예시:

```text
예상 대기 인원: 13명
예상 대기시간: 18분
기상 위험 점수: 82점

혼잡 점수 = 13*0.5 + 18*0.3 + 82*0.2
          = 6.5 + 5.4 + 16.4
          = 28.3점
```

정류장 공간이 좁거나 보행로 점유 위험이 높으면 공간 가중치를 추가합니다.

```text
congestionScore += sidewalkRiskBonus
```

### 8-3. 고도화 모델

위험도 분류 모델로 확장합니다.

고도화 단계에서는 단순 점수 합산 대신 과거 데이터를 학습합니다. 예를 들어 대기 인원 20명이라도 보행로가 넓은 정류장은 위험이 낮을 수 있고, 대기 인원 8명이라도 공간이 좁고 폭염이면 위험이 높을 수 있습니다. 모델은 정류장 공간 특성, 날씨, 대기시간, 미탑승 반복 여부를 함께 보고 위험 등급을 예측합니다.

예측 대상:

- 보행로 점유 위험
- 폭염·한파 대기 위험
- 현장 안내 필요 여부

## 9. Anomaly Detection AI

### 9-1. 목적

위치 조작, 반복 노쇼, 다중 체크인 등 서비스 공정성을 해치는 패턴을 감지합니다.

### 9-2. MVP 규칙

Anomaly Detection AI의 MVP는 명확한 규칙으로 시작합니다. 위치 인증 반경 밖에서는 체크인을 막고, 너무 빠른 위치 이동이나 반복 노쇼가 있으면 운영자에게 표시합니다. 초기에는 이용자 제재보다 서비스 공정성 유지와 운영자 확인용 신호로 쓰는 것이 적절합니다.

```text
if checkInDistance > 100m:
  체크인 불가

if userSpeedBetweenLocations > 120km/h:
  위치 이상 의심

if noShowCountLast7Days >= 3:
  반복 노쇼 경고

if sameDeviceMultipleAccountsInSameQueue:
  중복 체크인 의심
```

### 9-3. 고도화 모델

정상 행동과 다른 패턴을 찾는 비지도 이상탐지를 적용할 수 있습니다.

고도화 단계에서는 “정상 사용자 행동”과 다른 패턴을 자동으로 찾습니다. 예를 들어 대부분의 사용자는 정류장 근처에서 체크인하고 호출 후 일정 시간 안에 이동하지만, 어떤 계정은 여러 정류장에서 동시에 체크인하거나 매번 호출 후 나타나지 않을 수 있습니다. Isolation Forest 같은 이상탐지 모델은 이런 드문 패턴을 찾아냅니다.

후보:

- Isolation Forest
- One-Class SVM
- Local Outlier Factor

단, 개인정보 보호를 위해 개인의 정밀 이동 경로를 장기 저장하지 않고, 체크인 판정과 이상탐지에 필요한 최소 이벤트만 저장합니다.

## 10. Operator Insight AI

### 10-1. 목적

복잡한 예측 결과를 운영자가 바로 이해할 수 있는 문장으로 요약합니다.

### 10-2. MVP 방식

생성형 AI 없이 템플릿 기반으로 구현합니다.

Operator Insight AI의 MVP는 예측 결과를 미리 정해둔 문장 틀에 넣어 설명합니다. 예를 들어 위험 등급이 `위험`이고 미탑승이 반복되면 “다음차 안내와 운수사 피크 수요 알림이 필요합니다”라는 문장을 보여줍니다. 이 방식은 생성형 AI 비용이 없고, 잘못된 문장을 만들 위험이 낮습니다.

예:

```text
55305 이주택지 서울방향은 최근 2시간 평균 잔여좌석이 9.6석이고,
0석 신호가 반복되었습니다. 다음 접근 차량은 4개 정류장 전에서 23석이며,
중간 정류장 탑승 수요를 고려하면 보수 호출이 필요합니다.
```

미탑승 반복 시:

```text
08:46 이후 이주택지 서울방향에서 미탑승이 발생했고,
후속 차량에서도 대기 수요가 해소되지 않았습니다.
이번 시간대는 호출 인원을 보수적으로 잡고 다음차 안내를 함께 제공해야 합니다.
```

### 10-3. 고도화 방식

향후 OpenAI API 또는 사내 LLM을 이용해 운영 리포트를 자동 작성할 수 있습니다.

고도화 단계에서는 하루 동안의 정류장별 위험 신호, 미탑승 반복, 시간대별 수요 변화를 모아 운영 리포트를 자동 작성할 수 있습니다. LLM은 숫자 예측 자체를 담당하기보다, 이미 계산된 예측 결과를 사람이 읽기 쉬운 보고서 문장으로 바꾸는 역할을 맡는 것이 안전합니다.

입력:

- 정류장별 위험 지표
- 미탑승 반복 구간
- 시간대별 수요 변화
- 추천 조치

출력:

- 일일 운영 요약
- 운수사 배차 검토 리포트
- 지자체 민원 대응 보고서 초안


예상 질문: “AI 경험이 없는데 실제로 구현 가능한가?”

답변:

```text
초기 AI 전체를 단순 규칙으로만 시작하지 않습니다. 현재 미니PC에서 GBIS 데이터를 실시간 수집하고 있어 SeatFlow AI는 LightGBM, XGBoost, Random Forest 같은 경량 회귀 모델로 바로 학습할 수 있습니다. 이미 프로토타입에는 잔여좌석, 정류장 순번, 노쇼율, 실사 데이터를 이용한 탑승 가능 인원·호출 인원·미탑승 위험 계산 구조가 구현되어 있고, 다음 단계는 이 구조의 SeatFlow 부분을 실제 수집 데이터 기반 학습 모델로 교체하는 것입니다. 사용자 체크인 이벤트가 쌓이면 Boarding Probability와 Left-behind Risk도 분류 모델로 확장할 수 있습니다.
```

```text
단순 도착 알림은 버스가 언제 오는지만 알려줍니다. QueueBus는 차량이 목표 정류장에 도착했을 때 몇 명이 탈 수 있는지, 내 순번이면 이번 차를 탈 수 있는지, 호출 대상은 몇 번까지인지, 미탑승이 반복되는 정류장은 어디인지 예측합니다. 이 예측은 잔여좌석, 차량 위치, 정류장별 과거 탑승 패턴, 대기열, 노쇼율을 결합해야 하므로 AI 적용 가치가 있습니다.
```
