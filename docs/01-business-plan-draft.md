# QueueBus 사업계획서 초안

## 1. 사업 아이템 개요

QueueBus는 광역버스 정류장의 물리적 줄서기를 위치정보 기반 가상 대기열로 전환하고, 축적된 대기 수요 데이터를 AI로 분석해 호출 인원과 호출 타이밍을 최적화하는 대중교통 대기 관리 서비스입니다.

정식 제목 후보는 `QueueBus: AI 기반 위치인증형 광역버스 정류장 혼잡 예측 및 탑승 호출 서비스`입니다.

QueueBus는 좌석예약 서비스가 아닙니다. 정류장 현장에 실제로 도착한 승객의 대기 순서를 위치정보로 인증하고, AI로 탑승 수요와 호출 타이밍을 예측해 광역버스 정류장의 혼잡과 대기 불편을 줄이는 서비스입니다.

## 2. 문제 인식

수도권 광역버스 정류장은 출퇴근 시간대에 장시간 물리적 줄서기가 발생합니다. QueueBus 아이디어는 동탄에서 서울로 이동하며 4137번, 6002번, 4448번 등 여러 서울행 광역버스를 기다리는 과정에서 직접 체감한 불편에서 출발했습니다. 특히 폭염, 한파, 우천 상황에서는 정류장 주변에 오래 서 있는 행위 자체가 이용자 부담과 안전 위험으로 이어집니다.

현재 문제는 개인의 불편을 넘어 공공 교통 운영 문제로 확장됩니다.

- 장시간 야외 대기로 인한 폭염, 한파, 우천 노출
- 이번 버스를 탈 수 있는지 알기 어려운 탑승 가능성 불확실성
- 새치기, 순번 확인, 구두 질서 유지 과정에서 발생하는 이용자 간 갈등
- 긴 대기줄의 보행로 점유와 주변 상권/보행자 불편
- 지자체와 운수사가 정류장별, 노선별, 시간대별 실제 대기 수요를 정량적으로 파악하기 어려운 문제
- 민원과 체감에 의존하는 혼잡 정류장 관리
- 배차 조정, 현장 질서 관리, 안전 대응에 활용 가능한 데이터 부족

즉, 광역버스 정류장의 물리적 줄서기는 폭염·한파 노출, 보행로 점유, 새치기 갈등, 탑승 불확실성, 운영 데이터 부족 문제를 함께 만듭니다.

## 3. 해결 방안

QueueBus는 전체 대기자가 다시 줄을 서게 만드는 서비스가 아니라, 체크인 순서의 대기번호는 고정하고 내 앞 대기 인원만 실시간으로 갱신한 뒤, 이번 차량에 탑승 가능성이 높은 대기번호 구간만 기존 노선별 대기 위치로 짧게 호출하는 서비스입니다.

기본 흐름은 다음과 같습니다.

1. 사용자가 광역버스 정류장 근처에 도착합니다.
2. 앱이 사용자 GPS와 정류장 좌표를 비교해 정류장 반경 내 위치 인증을 수행합니다.
3. 사용자는 탈 노선을 선택하고 노선별 가상 대기열에 등록합니다.
4. 대기 순번은 위치 인증 완료 시점 기준 선착순으로 부여합니다.
5. AI는 현재 대기열, 차량별 잔여좌석, 버스 도착 정보, 시간대별 수요, 노쇼율, 날씨 데이터를 분석합니다.
6. AI는 이번 차량에 몇 명이 탈 수 있는지, 내 순번이면 이번 차량을 탈 수 있는지, 몇 명을 언제 호출해야 하는지를 예측합니다.
7. 버스 도착 전 탑승 가능성이 높은 대기번호 구간만 호출합니다.
8. 호출된 사용자는 앱에서 `내 앞 대기 인원`과 `이동할 기존 노선 대기 위치`를 확인합니다.
9. 사용자는 기존 바닥 버스번호가 표시된 해당 노선 대기 위치로 이동해 앞사람 수를 기준으로 짧게 정렬합니다.
10. 탑승 결과, 호출 응답, 노쇼, 대기시간 데이터가 축적되고 다음 예측에 활용됩니다.

QueueBus의 AI는 누가 먼저 탈지 정하지 않습니다. 순번은 위치 인증 기반 선착순 대기번호로 보장하며, 대기번호는 원칙적으로 바꾸지 않습니다. AI는 대신 이번 버스가 몇 명을 태울 수 있는지, 내 앞에 실제로 몇 명이 남아 있는지, 어느 대기번호 구간을 언제 기존 노선 대기 위치로 불러야 하는지를 예측합니다.

## 4. LBS 활용

QueueBus에서 위치정보는 부가 기능이 아니라 서비스 작동 조건입니다.

- 사용자 현재 위치: 정류장 근처에 실제로 도착했는지 판단
- 정류장 좌표: 지오펜싱 기준점
- 사용자 위치 이탈 여부: 대기열 유지, 경고, 취소 판단
- 버스 실시간 위치: 호출 시점 계산
- 차량의 노선 내 정류장 순번: 목표 정류장 도착 전 잔여좌석 예측
- 주변 대기 가능 공간: 그늘, 지하철 출구, 쉼터 등 분산 대기 안내

MVP에서는 사용자 좌표와 정류장 좌표의 거리를 Haversine 공식으로 계산하고, 100m 이내일 때 체크인을 허용합니다. 실제 GPS가 없는 발표 환경을 고려해 위치 인증 시뮬레이션 토글을 제공합니다.

향후에는 공식 버스정보 API, 정류장 좌표 DB, PostGIS 기반 공간 검색, 위치 이탈 감지, 정류장 주변 대기 가능 공간 추천 기능을 확장할 수 있습니다.

## 5. AI 활용

QueueBus의 AI는 공정한 순번 배정이 아니라 예측과 운영 최적화에 집중합니다.

- Demand Forecast AI: 정류장, 노선, 시간대별 대기 수요 예측
- SeatFlow AI: 차량별 목표 정류장 도착 시 예상 잔여좌석 예측
- Boarding Probability AI: 사용자별 다음 차량 탑승 가능성과 내 앞 대기 인원 예측
- Call Optimizer AI: 호출 인원, 호출 타이밍, 호출 대상 대기번호 구간 최적화
- Congestion & Risk AI: 보행로 점유, 폭염/한파, 반복 만차 위험 예측
- Anomaly Detection: 위치조작, 반복 노쇼, 다계정 중복 체크인 의심 패턴 탐지

초기 PoC에서는 규칙 기반과 이동평균 모델을 사용합니다. 이후 축적 데이터가 충분해지면 LightGBM, XGBoost, 시계열 예측 모델, Isolation Forest 기반 이상탐지 모델로 고도화합니다.

## 6. 차별성

QueueBus는 광역버스 좌석 예약 서비스가 아닙니다. 좌석을 사전에 확보하는 구조가 아니라, 정류장에 실제 도착한 승객의 현장 대기 순서를 위치정보로 인증하고 정류장 혼잡을 관리합니다.

또한 QueueBus는 단순 버스 도착 알림 서비스가 아닙니다. 도착 예정 시간을 알려주는 데서 끝나지 않고, 현재 가상 대기열과 예상 잔여좌석을 결합해 탑승 가능성, 내 앞 대기 인원, 호출 인원, 호출 타이밍, 호출 대상 대기번호 구간을 제공합니다.

QueueBus의 핵심 차별점은 다음입니다.

- 실제 정류장 반경 내 사용자만 체크인할 수 있는 위치 인증형 대기열
- 노선별 가상 대기열과 기존 노선 대기 위치의 연결
- 정류장별, 노선별, 시간대별 실제 대기 수요 데이터 축적
- AI 기반 호출 최적화와 혼잡 위험 예측
- 개인 이용자 편의와 지자체/운수사 운영 데이터를 동시에 제공하는 B2G/B2B 모델

## 7. 시장성

1차 사용자는 출퇴근 광역버스 이용자, 고령자, 임산부, 교통약자, 초행길 이용자, 폭염·한파 취약 이용자입니다. 이들은 물리적 줄서기의 불편, 탑승 가능성 불확실성, 현장 순번 갈등에 직접 노출됩니다.

2차 고객은 지자체, 운수사, 교통 운영기관, 정류장 관리기관입니다. 이들은 혼잡 정류장 관리, 민원 대응, 배차 개선, 현장 질서 유지, 안전 대책 수립에 필요한 정량 데이터를 필요로 합니다.

B2C 직접 과금보다 지자체 또는 운수사가 비용을 부담하는 B2G/B2B SaaS 모델이 적합합니다. 개인 이용자는 무료 또는 지자체 서비스로 제공하고, 운영기관은 정류장 단위 SaaS, 수요 분석 리포트, 현장 키트 설치, API 연동 비용을 지불하는 구조가 자연스럽습니다.

## 8. 기술 구현

MVP는 Vite, React, TypeScript 기반 웹 프로토타입으로 구현합니다. 별도 백엔드 없이 mock 데이터와 프론트엔드 예측 로직으로 발표 시연이 가능하게 구성합니다.

핵심 모듈은 다음과 같습니다.

- 위치 인증 모듈: Haversine 거리 계산, 100m 지오펜스 판정
- 대기열 모듈: 노선별 선착순 대기번호 발급, 내 앞 대기 인원 갱신
- SeatFlow 예측 모듈: 현재 잔여좌석, 남은 정류장 수, 평균 승하차 인원 기반 예상 탑승 가능 인원 계산
- Boarding Probability 모듈: 내 대기번호와 예상 탑승 가능 인원을 비교해 확률과 안내 문구 산정
- Call Optimizer 모듈: 노쇼율 보정 호출 인원과 호출 시점 계산
- Risk 모듈: 대기 인원, 예상 대기시간, 날씨 위험 점수 기반 혼잡 위험 등급 산정
- 운영자 대시보드: 집계 데이터, 차트, PoC 지표 표시

## 9. PoC 계획

초기 PoC는 혼잡한 광역버스 정류장 1곳과 노선 1~2개로 제한합니다.

1. 혼잡 광역버스 정류장 1곳 선정
2. 퇴근 시간대 핵심 노선 1~2개 선정
3. 차량별 잔여좌석, 도착 예정, 정류장 순번 데이터를 mock 또는 공식 API 연동 가능 형태로 정리
4. 현장 관찰 1회로 실제 대기 인원, 탑승 인원, 미탑승 인원, 보행로 점유 여부 확인
5. 웹 기반 프로토타입으로 위치 인증, 대기번호, 내 앞 대기 인원, 호출 대상 대기번호 구간 구현
6. 관리자 대시보드로 정류장별·노선별·시간대별 혼잡 데이터 표시
7. 초기 규칙 기반 예측 결과와 현장 관찰 결과를 비교해 개선 항목 도출

PoC 검증 지표는 위치 인증 성공률, 가상 대기열 등록 완료율, 호출 응답률, 노쇼율, 탑승 가능성 예측 정확도, 평균 물리적 줄서기 시간 감소, 보행로 점유 완화 여부, 사용자 만족도입니다.

## 10. 사업화 모델

- 지자체 PoC 구축비
- 혼잡 정류장 단위 SaaS 이용료
- 운수사 대상 노선별 대기 수요 대시보드 구독
- 정류장별·시간대별 수요 분석 리포트
- QR 안내판, 기존 바닥 버스번호 연계 호출 안내, 전광판 연동 등 현장 키트 설치비
- 향후 교통 운영 시스템 API 연동 및 유지보수 비용

초기 시장 진입은 민원이 잦고 물리적 대기줄이 긴 정류장 1~3곳의 PoC로 시작합니다. PoC 성과가 확인되면 같은 지자체 내 주요 광역버스 정류장으로 확장하고, 이후 운수사/광역교통 운영기관 대상 대시보드 구독 모델로 확대합니다.

## 11. 개인정보 및 위치정보 보호

QueueBus는 위치정보를 정류장 체크인, 대기열 유지, 호출 안내 목적에 한정해 사용합니다.

- 개인의 정밀 이동 경로를 장기 저장하지 않습니다.
- 운영자 대시보드에는 개인 단위 위치가 아니라 집계 데이터만 제공합니다.
- 탑승 완료 후 개인 식별 가능한 위치정보는 최소 보관하고, 통계 목적 데이터는 비식별화합니다.
- 위치정보 수집·이용 목적과 보관 기간을 앱에서 명확히 안내합니다.
- 위치 이탈 감지와 이상탐지는 서비스 공정성 유지 목적에 한정합니다.
- 위치정보사업 또는 위치기반서비스 관련 인허가/신고 필요 여부를 사업화 단계에서 검토합니다.

## 12. 기대효과

이용자 측면에서는 정류장 앞 장시간 줄서기 부담을 줄이고, 탑승 가능성 정보를 통해 대기 불안을 낮출 수 있습니다. 고령자, 임산부, 교통약자, 폭염·한파 취약 이용자에게 특히 효과가 큽니다.

운영기관 측면에서는 정류장별·노선별·시간대별 대기 수요 데이터가 축적됩니다. 이 데이터는 혼잡 정류장 지정, 배차 개선, 현장 질서 관리, 민원 대응, 안전 대책 수립의 근거가 됩니다.

사회적 측면에서는 보행로 점유 완화, 폭염·한파 노출 감소, 이용자 간 갈등 감소, 대중교통 이용 경험 개선에 기여합니다.

QueueBus는 사용자가 줄서기를 할수록 정류장별·노선별·시간대별 탑승 수요 데이터가 축적되고, 이 데이터는 AI 기반 혼잡 예측과 배차·정류장 운영 개선의 근거가 됩니다.

## 14. 팀 역량

팀은 위치기반 서비스 기획, 프론트엔드 프로토타입 구현, 데이터 기반 PoC 설계 역량을 중심으로 구성합니다. 초기에는 사업기획/사용자 리서치 담당, 프론트엔드/데이터 구현 담당, 교통 데이터/운영 협력 담당으로 역할을 나눕니다.

팀 역량 보완이 필요한 영역은 공식 버스정보 API 연동, 위치정보 법제 검토, 지자체/운수사 협력, 현장 실증 운영입니다. 공모전의 BM 컨설팅, 기술 멘토링, AI 전환 컨설팅과 연계해 해당 영역을 보완합니다.
