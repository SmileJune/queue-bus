# QueueBus AI MVP 기능 설계

작성일: 2026-06-02  
목적: AI 개발 경험이 부족하다는 우려를 줄이기 위해, 공모전 제출·발표 전에 실제로 구현 가능한 AI MVP 범위를 정의합니다.

## 1. 설계 원칙

QueueBus의 AI MVP는 처음부터 모든 모듈을 복잡한 모델로 만들 필요는 없지만, 핵심 기능인 SeatFlow AI는 현재 쌓인 GBIS 데이터를 이용해 경량 학습 모델로 시작하는 것이 더 설득력 있습니다. 심사위원에게 보여줘야 하는 것은 “AI라는 이름만 붙인 아이디어”가 아니라, 실제 수집 데이터로 학습한 예측 결과와 그 결과를 이용한 호출·위험 판단입니다.

따라서 MVP는 다음 원칙으로 설계합니다.

| 원칙 | 설명 |
| --- | --- |
| 핵심 모델은 학습형으로 시작 | GBIS 잔여좌석·차량 위치 데이터가 충분한 SeatFlow AI는 LightGBM/XGBoost급 경량 회귀 모델로 시작 |
| 데이터 부족 영역은 규칙 기반 보조 | 사용자 체크인·호출·노쇼 데이터가 아직 부족한 기능은 설명 가능한 규칙으로 시작 |
| 실제 데이터 반영 | GBIS 수집 데이터와 55305 이주택지 현장 실사 데이터를 mock이 아닌 근거 데이터로 사용 |
| 결과를 화면화 | 사용자 화면, 운영자 대시보드, AI 예측 탭에서 같은 결과를 다른 관점으로 표시 |
| 오차를 측정 | 예측값과 실제 탑승·미탑승 결과를 비교해 “학습 가능한 구조”를 증명 |
| 단계적 고도화 | MVP는 SeatFlow 학습 모델, PoC는 탑승확률·미탑승 분류 모델, 사업화는 시계열·이상탐지·LLM 리포트로 확장 |

## 2. 공모전에서 보여줄 AI 기능 5개

### 2-1. SeatFlow AI: 차량별 탑승 가능 인원 예측

목표: 현재 차량 잔여좌석과 목표 정류장 전 승하차 패턴을 이용해, 해당 정류장에서 몇 명이 탈 수 있을지 예측합니다.

입력:

| 입력값 | 예 |
| --- | --- |
| 현재 잔여좌석 | 2석 |
| 차량 현재 정류장 순번 | 신안2차/반도4차 |
| 목표 정류장 | 55305 이주택지 |
| 정류장 사이 예상 탑승 | 0명 |
| 정류장 사이 예상 하차 | 0명 |
| 목표 정류장 하차 관측 또는 추정 | 1명 |

MVP 계산:

```text
목표 정류장 예상 탑승 가능 인원 =
  현재 잔여좌석
  - 목표 정류장 전 예상 탑승
  + 목표 정류장 전 예상 하차
  + 목표 정류장 도착 후 하차 추정
```

오늘 실사 예시:

```text
차량 1040
이주택지 도착 전 빈자리 2석
이주택지 하차 1명
실제 탑승 3명
미탑승 4명
```

설명 포인트:

잔여좌석 2석만 보면 2명 탑승으로 예측하지만, 현장 하차 1명을 반영하면 실제 탑승 가능 인원 3명을 설명할 수 있습니다. 이 기능은 GBIS 데이터와 현장 실사를 결합해야 하는 이유를 보여줍니다.

현재 구현 위치:

- `src/lib/prediction.ts`의 `calculateExpectedRemainingSeats`
- `src/lib/prediction.ts`의 `estimateBoardedCountFromSeatDelta`
- `src/App.tsx`의 `AiPanel`, `OperatorDashboard`

### 2-2. Boarding Probability AI: 내 순번 탑승 가능성

목표: 사용자의 대기번호와 예상 탑승 가능 인원을 비교해 이번 차량 탑승 가능성을 안내합니다.

입력:

| 입력값 | 예 |
| --- | --- |
| 내 대기번호 | 9번 |
| 내 앞 대기 인원 | 8명 |
| 예상 탑승 가능 인원 | 3명 |
| 예상 노쇼율 | 20% |
| 다음 차량 배차 간격 | 9분 |

MVP 계산:

```text
유효 순번 = 내 대기번호 - 내 앞 사람 예상 노쇼 수
이번 차량 확률 = 예상 탑승 가능 인원과 유효 순번의 차이로 산정
다음 차량 확률 = 이번 차량 확률 + 배차 보정
```

사용자 문구:

```text
이번 차량 탑승 가능성 낮음
현재 순번으로는 다음 차량 대기 가능성이 높습니다.
호출 전까지 정류장 앞에 줄 서지 않아도 됩니다.
```

현재 구현 위치:

- `src/lib/prediction.ts`의 `calculateBoardingProbability`
- 사용자 화면의 탑승 가능성 링

### 2-3. Call Optimizer AI: 호출 인원·호출 시점 계산

목표: 버스 도착 직전 전체 대기자가 한 번에 몰리지 않도록, 이번 차량에 탈 가능성이 높은 대기번호 구간만 호출합니다.

입력:

| 입력값 | 예 |
| --- | --- |
| 예상 탑승 가능 인원 | 3명 |
| 노쇼율 | 20% |
| 최대 호출 인원 | 10명 |
| 정류장까지 걸어오는 시간 | 90초 |
| 정렬 버퍼 | 30초 |

MVP 계산:

```text
호출 인원 = min(최대 호출 인원, ceil(예상 탑승 가능 인원 / (1 - 노쇼율)))
호출 시점 = 버스 도착 120초 전
```

사용자 문구:

```text
대기번호 1~4번 호출
도착 2분 전 기존 노선 대기 위치로 이동하세요.
```

운영자 문구:

```text
호출 구간을 짧게 유지하세요.
미호출자는 분산 대기를 안내합니다.
```

현재 구현 위치:

- `src/lib/prediction.ts`의 `calculateCallCount`
- `src/App.tsx`의 `CallRangePanel`

### 2-4. Left-behind Detector: 미탑승 위험 감지

목표: 대기자가 실제로 버스에 타지 못한 상황을 운영자에게 알립니다. 공모전에서는 이 기능이 “운수사·지자체가 돈을 낼 이유”를 가장 잘 보여줍니다.

입력:

| 입력값 | 예 |
| --- | --- |
| 도착 전 대기 인원 | 7명 |
| 하차 인원 | 1명 |
| 탑승 인원 | 3명 |
| 남은 대기 인원 | 4명 |
| 다음 차량에서도 남은 인원 | 4명 |

MVP 계산:

```text
미탑승 인원 = max(도착 전 대기 인원 - 탑승 인원, 0)
미해소 대기 = 다음 차량 이후에도 남은 인원
위험 플래그 = 미탑승 인원 > 0 또는 미해소 대기 2회 이상 지속
```

오늘 실사 예시:

| 차량 | 대기 | 하차 | 탑승 | 미탑승/잔여 |
| --- | ---: | ---: | ---: | ---: |
| 1040 | 7 | 1 | 3 | 4 |
| 1043 | 5 | 1 | 1 | 4 |
| 4811 | 3 | 0 | 0 | 3 |

운영자 알림:

```text
55305 이주택지 서울방향에서 08:46 이후 미탑승이 연속 발생했습니다.
다음 배차 또는 현장 안내 필요성이 높습니다.
```

현재 상태:

- CSV 원자료는 `data/field-observation-2026-06-02.csv`에 있음
- MVP에는 아직 전용 감지 함수가 없으므로 추가 구현하기 좋음

추천 구현:

```ts
function detectLeftBehindRisk(events) {
  const unresolved = events.filter((event) => event.failedBoardingCount > 0);
  const repeated = unresolved.length >= 2;

  return {
    riskLevel: repeated ? "위험" : unresolved.length === 1 ? "높음" : "보통",
    message: repeated
      ? "미탑승이 후속 차량에서도 해소되지 않았습니다."
      : "미탑승 발생 차량이 있습니다.",
  };
}
```

### 2-5. Operator Insight AI: 운영자용 한 줄 설명 생성

목표: 계산 결과를 운영자가 바로 이해할 수 있는 문장으로 변환합니다. 이 기능은 생성형 AI를 쓰지 않아도 템플릿 기반으로 구현할 수 있습니다.

입력:

| 입력값 | 예 |
| --- | --- |
| 정류장 | 55305 이주택지 |
| 방향 | 서울방향 |
| 미탑승 발생 시각 | 08:46 |
| 반복 여부 | 후속 차량에서도 미해소 |
| 권장 조치 | 다음차 안내, 호출 인원 축소, 배차 검토 |

출력:

```text
08:46 이후 55305 이주택지 서울방향에서 미탑승이 발생했고,
후속 차량에서도 대기 수요가 해소되지 않았습니다.
이번 시간대는 호출 인원을 보수적으로 잡고 다음차 안내를 함께 제공해야 합니다.
```

확장 방향:

- MVP: 템플릿 문장
- PoC: 규칙 기반 추천
- 고도화: OpenAI API 또는 자체 LLM으로 운영 리포트 자동 요약

## 4. 데이터 설계

| 탑승 인원 | 현장 실사 또는 탑승 확인 이벤트 | 예측 오차 계산 |

### 이벤트 로그

향후 MVP에는 다음 이벤트 로그 구조를 추가하면 됩니다.

```text
check_in

call_acknowledged
arrived_at_queue
boarded
left_behind

```

이 이벤트가 쌓이면 단순한 발표용 화면이 아니라 실제 AI 학습 데이터 파이프라인이 됩니다.

## 5. 예측 오차 측정

AI 경험 부족 우려를 줄이려면 “모델을 만들었다”보다 “오차를 측정하고 개선할 구조가 있다”를 보여줘야 합니다.

MVP에서 바로 보여줄 수 있는 평가 지표:

| 지표 | 계산 |
| --- | --- |
| 탑승 가능 인원 오차 | `abs(예상 탑승 가능 인원 - 실제 탑승 인원)` |
| 미탑승 감지 정확도 | 실제 미탑승 발생 여부와 예측 위험 플래그 비교 |
| 호출 과다 | 호출 인원 - 실제 탑승 인원 |
| 호출 부족 | 실제 탑승 가능 인원 - 호출 인원 |
| 다음차 안내 적중 | 다음차 안내 후 실제 미탑승 지속 여부 |

오늘 실사 기반 예:

```text
1040 차량
예상 빈자리 기반 탑승 가능: 2명
하차 반영 탑승 가능: 3명
실제 탑승: 3명
미탑승: 4명
```

이 사례는 “하차 인원 보정이 예측 정확도를 높인다”는 구체적 근거가 됩니다.

## 6. 구현 우선순위

### P0: 제출 전 바로 보여줄 기능

1. 기존 AI 예측 탭에 “실사 검증 사례” 카드 추가
2. `data/field-observation-2026-06-02.csv`를 mock 데이터로 읽어 표출
3. 1040 차량의 예측값과 실제값 비교 표시
4. 미탑승 위험 감지 결과 표시
5. 운영자 대시보드에 “미탑승 반복 알림” 추가

### P1: 발표 전 완성하면 좋은 기능

1. 사용자가 노선/정류장을 바꾸면 예측값이 함께 바뀌는 시나리오
2. 호출 인원을 조절했을 때 물리적 줄서기 시간이 줄어드는 시뮬레이션
3. 노쇼율 슬라이더 또는 시나리오 토글
4. 기상 위험 점수에 따른 호출 정책 변화

### P2: 선정 후 PoC 기능

1. 실제 GBIS 데이터 자동 반영
2. 현장 체크인 이벤트 수집
3. 예측 오차 리포트 자동 생성
4. 정류장·시간대별 반복 미탑승 지도화
5. 운영자 리포트 자동 요약

QueueBus의 AI는 초기 단계부터 핵심 기능인 SeatFlow AI에 경량 학습 모델을 적용한다. 현재 미니PC에서 수집 중인 GBIS 잔여좌석·차량 위치 데이터와 좌석 변화 기반 탑승 추정치를 이용해 목표 정류장 도착 시 예상 잔여좌석을 예측한다. 사용자 체크인·호출·노쇼 데이터가 아직 부족한 Boarding Probability, Call Optimizer, Left-behind Risk는 규칙 기반으로 시작하되, PoC에서 이벤트 로그가 쌓이면 분류 모델과 정책 최적화 모델로 전환한다. 2026년 6월 2일 55305 이주택지 현장 실사에서는 차량 1040번의 도착 시 빈자리 2석, 하차 1명, 실제 탑승 3명, 미탑승 4명 사례를 확인했으며, 이 데이터는 잔여좌석 기반 예측에 하차·대기열 데이터를 결합해야 함을 보여준다.
