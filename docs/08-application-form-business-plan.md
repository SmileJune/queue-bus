## Ⅰ. 기업가 정신

### 1. 추진역량

#### 1-1. 서비스/제품 아이디어의 개발 동기 및 배경

QueueBus는 출퇴근 시간대 광역버스 정류장에서 반복되는 물리적 줄서기 문제를 직접 경험한 데서 출발했습니다. 특히 동탄에서 서울로 이동할 때 4137번, 6002번, 4448번 등 여러 서울행 광역버스를 기다리며, 긴 줄을 서고도 이번 차량을 탈 수 있는지 알기 어렵고 노선별 대기 순서가 현장에서 명확히 관리되지 않는 불편을 반복적으로 체감했습니다. 폭염·한파·우천 상황에서는 야외 대기 자체가 큰 부담이 되며, 대기줄이 인도를 길게 점유하면서 보행자 불편이 발생하고, 새치기나 순번 확인 과정에서 이용자 간 갈등이 생깁니다.

이 문제는 개인의 불편을 넘어 정류장 운영 문제입니다. 지자체와 운수사는 정류장별·노선별·시간대별 실제 대기 인원, 승차 가능성 변동, 대기열 길이, 미탑승 인원 같은 현장 수요 데이터를 정량적으로 파악하기 어렵습니다. 그 결과 혼잡 정류장 관리가 민원과 체감에 의존하고, 배차 개선이나 현장 질서 관리의 근거 데이터가 부족합니다.

QueueBus는 이러한 오프라인 줄서기 문제를 위치정보 기반 가상 대기열로 전환합니다. 정류장 근처에 실제 도착한 사용자만 위치 인증 후 노선별 대기열에 등록하고, 사용자는 줄에 계속 서는 대신 고정 대기번호와 `내 앞 대기 인원`을 보며 주변 공간에서 기다릴 수 있습니다. AI는 이번 버스와 다음 버스의 승차 가능성, 호출할 대기번호 구간, 호출 타이밍을 예측해 필요한 인원만 짧게 호출합니다.

#### 1-2. 서비스/제품의 필요성 및 창의성

QueueBus는 좌석예약 서비스가 아닙니다. 정류장 현장에 실제로 도착한 승객의 대기 순서를 위치정보로 인증하고, 줄 대신 대기번호로 편하게 기다리게 하며, AI로 다음 버스 승차 가능성과 호출 타이밍을 예측해 광역버스 정류장의 혼잡과 대기 불편을 줄이는 서비스입니다.

필요성은 다음 다섯 가지로 정리됩니다.

- 이용자 편의: 장시간 물리적 줄서기를 줄이고, 호출 전에는 주변 그늘·쉼터·실내 공간에서 기다릴 수 있게 합니다.
- 승차 가능성 판단: 대기열 순번, 차량 잔여좌석, 앞순번 노쇼 가능성을 결합해 이번 버스와 다음 버스의 승차 가능성을 안내합니다.
- 안전·복지: 폭염·한파·우천 상황에서 야외 대기 노출을 줄이며, 고령자·임산부·교통약자의 대기 부담을 완화합니다.
- 질서 유지: 위치 인증 기반 선착순 대기번호는 고정하고, 내 앞 대기 인원과 호출 대상 대기번호 구간을 안내해 새치기와 구두 순번 확인 갈등을 줄입니다.
- 운영 수요 신호: 정류장별·노선별·시간대별 대기 수요, 호출 응답률, 노쇼율, 미탑승 인원 데이터를 축적해 운수사의 배차 간격 조정, 예비차 투입 검토, 지자체의 현장 운영 개선 근거를 제공합니다.

창의성은 `위치 인증형 현장 대기열`, `다음 버스 승차 가능성 예측`, `운수사 운영 수요 신호`를 결합한 점입니다. 기존 버스 도착 알림은 버스가 언제 오는지를 알려주는 데 그치고, 좌석예약은 특정 차량의 좌석 확보에 초점을 둡니다. QueueBus는 정류장 현장에 실제 도착한 승객의 대기번호를 고정 관리하고, 호출 전에는 물리적 줄서기 부담을 줄이며, 전체 대기자가 아닌 호출 대상 대기번호 구간만 기존 노선 대기 위치로 이동시키는 정류장 대기 관리 서비스입니다.

QueueBus의 AI는 누가 먼저 탈지 정하지 않습니다. 순번은 위치 인증 기반 선착순 대기번호로 보장하고, 앞사람 변동은 `내 앞 대기 인원`으로 안내합니다. AI는 대신 이번 버스가 몇 명을 태울 수 있는지, 내 순번이면 이번 버스 또는 다음 버스를 탈 가능성이 어느 정도인지, 어느 대기번호 구간을 언제 기존 노선 대기 위치로 불러야 하는지, 운수사에 어떤 시간대 수요 신호를 보내야 하는지를 예측합니다.

현장 관찰 전 예비 근거로 M4137 노선의 경기도 GBIS 공식 API 데이터를 수집했습니다. 2026-05-28 13:47부터 2026-05-30 20:18까지 차량별 잔여좌석, 정류장 순번, 도착 예측 정보를 수집한 결과, 퇴근 방향인 `서울→동탄` 6개 대상 정류장은 3일 관측과 정류장별 30개 이상 샘플 기준을 충족했습니다. 이 데이터의 목적은 `0석 발생` 자체를 강조하는 것이 아니라, 사용자에게 다음 버스 승차 가능성을 안내하고 운영기관에 시간대별 대기 수요 신호를 제공할 수 있는지를 검증하는 것입니다.

## Ⅱ. 추진 활동

### 2. 시장성

#### 2-1. 사업모델의 시장성

QueueBus의 1차 사용자는 출퇴근 광역버스 이용자입니다. 특히 혼잡 정류장에서 장시간 줄을 서는 직장인, 고령자, 임산부, 교통약자, 초행길 이용자, 폭염·한파 취약 이용자가 직접적인 수혜자입니다. 이들은 매일 반복되는 대기 불편과 탑승 가능성 불확실성을 겪기 때문에 서비스 반복 사용 가능성이 높습니다.

2차 고객은 지자체, 운수사, 교통 운영기관, 정류장 관리기관입니다. 이들은 혼잡 정류장 관리, 민원 대응, 배차 개선, 현장 질서 유지, 폭염·한파 안전 대응을 위해 실제 대기 수요 데이터가 필요합니다. QueueBus는 사용자 앱을 통해 대기열 이벤트를 수집하고, 운영자 대시보드에서 정류장별·노선별·시간대별 대기 수요와 승차 가능성 신호를 집계해 제공합니다. 운수사는 이를 통해 특정 시간대 배차 간격 조정, 예비차 투입 검토, 기사·현장 안내 인력 배치의 판단 근거를 얻을 수 있습니다.

사업모델은 B2C 직접 과금보다 B2G/B2B SaaS가 적합합니다. 개인 이용자는 무료 또는 지자체 서비스로 제공하고, 운영기관이 정류장 단위 SaaS 이용료, 대기 수요 리포트, 운수사 운영 알림 API, 현장 키트 설치비, API 연동 비용을 부담하는 구조입니다.

초기 시장 진입은 전국 단위가 아니라 혼잡 광역버스 정류장 1곳과 노선 1~2개에서 PoC를 수행하는 방식이 적절합니다. 한 정류장에서 대기시간 감소, 호출 응답률, 노쇼율, 보행로 점유 완화, 예측 정확도 같은 지표를 확보한 뒤 같은 지자체 내 주요 정류장으로 확장합니다.

- 대기 순번은 AI가 아니라 위치 인증 완료 시점 기준 선착순으로 부여해 공정성을 보장합니다.
- AI는 예측과 운영 최적화에만 사용하므로 설명 가능성과 수용성이 높습니다.
- 고정 대기번호, 내 앞 대기 인원, 다음 버스 승차 가능성, 기존 노선 대기 위치를 연결해 버스 도착 직전 재혼잡 문제를 줄입니다.
- 운수사와 운영기관이 활용 가능한 대기 수요와 배차 판단 데이터를 축적합니다.

### 3. 성장성

#### 3-1. 사업 추진 목표 및 추진 계획

QueueBus의 단기 목표는 공모전 기간 내 발표 가능한 웹 기반 PoC 프로토타입과 제출용 사업계획서를 완성하는 것입니다. MVP는 실제 상용 서비스가 아니라 `위치 인증 → 고정 대기번호 → 내 앞 대기 인원 → 탑승 가능성 → 호출 대상 대기번호 구간 → 운영자 대시보드` 흐름을 심사위원이 이해할 수 있도록 구현합니다.

중기 목표는 혼잡 광역버스 정류장 1곳에서 제한적 PoC를 수행하는 것입니다. 퇴근 피크 시간대 현장 관찰로 실제 대기 인원, 탑승 인원, 미탑승 인원, 줄 길이, 보행로 점유 여부를 수집하고, QueueBus 예측 로직과 비교합니다.

장기 목표는 지자체·운수사 대상 정류장 단위 SaaS로 확장하는 것입니다. 대시보드와 리포트 기능을 통해 혼잡 정류장 지정, 배차 간격 조정 검토, 예비차 투입 판단, 현장 안내 인력 배치, 폭염·한파 안전 대응의 의사결정 근거를 제공합니다.

| 단계 | 기간 | 목표 | 핵심 산출물 |
| --- | --- | --- | --- |
| MVP | 1개월 | 발표용 PoC 구현 | React 프로토타입, mock 데이터, 예측 로직 |
| 현장 PoC | 2~3개월 | 정류장 1곳, 노선 1~2개 실증 | 관찰 데이터, 예측 오차, 사용자 피드백 |
| 시범 운영 | 4~6개월 | 지자체/운수사 협력 | 관리자 대시보드, 기존 노선 대기 위치 연계 호출 안내 |
| 사업화 | 1년 | 정류장 단위 SaaS | 구독형 대시보드, 정기 리포트, 운수사 운영 알림 API |

최종 성과목표는 다음과 같습니다.

- PoC 정류장 기준 평균 물리적 줄서기 시간 30% 이상 감소
- 호출 응답률 80% 이상
- 차량별 탑승 가능 인원 예측 오차 ±3명 이내
- 반복 혼잡 시간대 운영자 알림 정확도 70% 이상
- 정류장별·노선별·시간대별 대기 수요 리포트 자동 생성
- 운수사 배차 간격 조정 검토에 활용 가능한 피크 수요 알림 제공

사회 문제 해결 및 ESG 요소는 다음과 같습니다.

- 환경·안전: 폭염·한파·우천 상황에서 야외 대기 시간을 줄여 시민 안전을 높입니다.
- 사회: 고령자, 임산부, 교통약자 등 대기 취약 이용자의 부담을 줄입니다.
- 거버넌스: 민원과 체감이 아닌 데이터 기반 정류장 운영을 가능하게 합니다.
- 도시 보행권: 긴 대기줄의 인도 점유를 줄여 보행자 불편을 완화합니다.

### 4. 기술성

#### 4-1. 사업모델 구현 방안 및 기술력

QueueBus MVP는 Vite, React, TypeScript 기반 웹 프로토타입으로 구현합니다. 별도 백엔드 없이 mock CSV와 프론트엔드 예측 로직으로 공모전 발표 시연이 가능하도록 구성했습니다. 향후 실제 PoC에서는 FastAPI 또는 Spring Boot 기반 API 서버, PostGIS 기반 정류장 지오펜스, 공식 버스정보 API, Redis 기반 대기열 저장소를 단계적으로 도입할 수 있습니다.

핵심 구현 구조는 다음과 같습니다.

| 모듈 | MVP 구현 | 고도화 방향 |
| --- | --- | --- |
| 위치 인증 | 사용자 좌표와 정류장 좌표의 Haversine 거리 계산, 100m 이내 체크인 | PostGIS, 위치 이탈 감지, GPS 정확도 보정 |
| 대기열 | 노선별 선착순 대기번호 발급, 내 앞 대기 인원 갱신 | 실시간 동기화, 운영자 개입 로그 |
| SeatFlow 예측 | 현재 잔여좌석, 남은 정류장 수, 평균 승하차 인원 기반 휴리스틱 | LightGBM/XGBoost 회귀 모델 |
| 탑승 가능성 | 내 대기번호, 예상 잔여좌석, 노쇼율 기반 확률 안내 | 로지스틱 회귀, 확률 보정 모델 |
| 호출 최적화 | 예상 탑승 가능 인원과 노쇼율 기반 호출 인원 계산 | 실시간 응답률 기반 예비 호출 |
| 운영 수요 신호 | 시간대별 대기열 길이, 미탑승 추정, 반복 저좌석 신호 표시 | 배차 간격 조정 추천, 예비차 투입 알림 |
| 혼잡 위험 | 대기 인원, 예상 대기시간, 날씨 위험 점수 기반 등급 | 시계열 혼잡 예측, 위험 이벤트 분류 |
| 이상탐지 | 위치 이탈, 반복 노쇼, GPS 순간이동 규칙 탐지 | Isolation Forest 등 비지도 이상탐지 |

위치정보 기술 적용 방식은 다음과 같습니다.

- 사용자 현재 위치를 정류장 체크인 목적에 한정해 사용합니다.
- 정류장 좌표를 기준으로 100m 반경 지오펜스를 구성합니다.
- 사용자가 정류장 반경 밖으로 일정 시간 이상 이탈하면 재인증 또는 대기열 유지 경고를 제공합니다.
- 버스 실시간 위치와 정류장 순번을 활용해 호출 시점을 계산합니다.
- 주변 그늘, 쉼터, 지하철 출구 등 분산 대기 공간 안내로 확장할 수 있습니다.

MVP 예측 로직은 다음과 같이 설명 가능합니다.

```text
expectedRemainingSeats =
  currentRemainingSeats
  - averageBoardingPerStop * remainingStopsToTarget
  + averageAlightingPerStop * remainingStopsToTarget

callCount =
  min(maxCallCount, ceil(expectedRemainingSeats / (1 - noShowRate)))

congestionScore =
  waitingCount * 0.5
  + expectedWaitMinutes * 0.3
  + weatherRiskScore * 0.2
```

개인정보 및 위치정보 보호 원칙은 다음과 같습니다.

- 위치정보는 정류장 체크인, 대기열 유지, 호출 안내 목적에 한정합니다.
- 개인의 정밀 이동 경로를 장기 저장하지 않습니다.
- 운영자 대시보드에는 개인 단위 위치가 아니라 집계 데이터만 제공합니다.
- 탑승 완료 후 개인 식별 가능한 위치정보는 최소 보관합니다.
- 위치정보 수집·이용 목적과 보관 기간을 명확히 안내합니다.
- 이상탐지는 서비스 공정성 유지 목적에 한정합니다.

## 최종 HWP 작성 메모

- 분량은 HWP 기준 20페이지 이내로 유지합니다.
