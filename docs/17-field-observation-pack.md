# 현장조사 실행 패키지

목적: 현장조사 당일에는 관찰에만 집중할 수 있도록, 조사 위치·시간·기록 항목·사진 목록을 미리 고정합니다. 현장조사의 초점은 `0석 확인`이 아니라, 이용자가 왜 계속 줄을 설 수밖에 없는지, 대기번호 기반 분산 대기가 어떤 불편을 줄일 수 있는지, 운수사에 제공할 대기 수요 신호가 실제 현장과 맞는지를 확인하는 것입니다.

## 1. 최우선 관찰 지점

| 우선순위 | 정류장 | 방향 | 권장 시간 | 데이터상 이유 | 현장 확인 질문 |
| --- | --- | --- | --- | --- | --- |
| P0 | 명동입구 | 서울→동탄 M4137 | 평일 16:30~18:45 | 승차 가능성 변동이 크고 다음차 안내가 필요한 구간 | 실제 대기자가 줄 대신 대기번호로 기다릴 여지가 있는가 |
| P0 | 명동성당 | 서울→동탄 M4137 | 평일 17:30~18:45 | 후반 정류장이라 승차 가능성 안내 가치가 큰 구간 | 대기자가 적어 보여도 탑승 불확실성이 큰가 |
| P1 | 서울역버스환승센터 6번승강장 | 서울→동탄 M4137 | 평일 16:45~17:45 | 좌석 여유가 시간대별로 급변하는 구간 | 호출 인원 조정과 다음차 안내가 필요한가 |
| P1 | 한신더휴 | 동탄→서울 M4137 | 주말 10:30~12:00 | 휴일 외출 수요가 누적되는 구간 | 주말 점심 외출 수요도 줄서기 문제인가 |
| P2 | 이주택지.상록.경남아파트 | 동탄→서울 M4137 | 주말 10:15~13:30 | 후반 정류장 탑승 가능성 안내가 필요한 구간 | 후반 정류장에서 다음차 안내 수요가 반복되는가 |

혼자 조사한다면 평일 퇴근 `명동입구`를 먼저 봅니다. 공모전 제출 관점에서는 “실제 줄 선 사람 수, 호출 전 분산 대기 가능 공간, 탑승 인원, 미탑승 인원”이 가장 강한 보완 근거이기 때문입니다.

## 2. 최소 관찰 기준

| 항목 | 기준 |
| --- | --- |
| 관찰 시간 | 한 지점 90분 이상 |
| 차량 수 | 같은 노선 6대 이상 권장 |
| 기록 단위 | 차량 1대 도착마다 1행 |
| 필수 수치 | 도착 전 대기 인원, 탑승 인원, 하차 인원, 미탑승 인원 |
| 보조 수치 | 줄 길이, 보행로 점유 여부, 주변 대기 가능 공간, 날씨, 특이사항 |
| 사진 | 개인 식별이 어려운 거리에서 3~5장 |

시간이 부족하면 `대기 인원`, `탑승 인원`, `미탑승 인원`, `버스 출발 시각`만이라도 정확히 기록합니다.

## 3. 기록 타이밍

| 시점 | 기록 |
| --- | --- |
| 버스 도착 3~5분 전 | 현재 대기 인원, 줄 길이, 보행로 점유 여부 |
| 버스 문 열림 | 도착 시각, 차량 식별 메모, 도착 전 잔여좌석 |
| 탑승 중 | 탑승 인원, 하차 인원 |
| 버스 출발 직후 | 출발 시각, 남은 대기 인원, 못 탄 인원, 출발 후 잔여좌석 |
| 다음 차량 전 | 새로 합류한 대기자 변화, 줄 재형성 여부, 대기번호 안내가 있었다면 줄을 줄일 수 있었을 공간 |

잔여좌석은 현장에서 앱/전광판/GBIS 대시보드로 확인할 수 있는 경우에만 적고, 확인이 어려우면 비워둡니다.

## 4. 사진 체크리스트

| 사진 | 목적 | 주의 |
| --- | --- | --- |
| 정류장 전체 전경 | 공간 제약과 줄 위치 설명 | 얼굴이 식별되지 않는 거리에서 촬영 |
| 줄 끝이 보이는 장면 | 물리적 줄서기 부담 근거 | 특정 개인을 클로즈업하지 않음 |
| 바닥 노선번호 또는 승강장 표식 | QueueBus 호출 위치 UX 설명 | 노선번호/표식 위주로 촬영 |
| 주변 대기 가능 공간 | 호출 전 분산 대기 UX 설명 | 영업장 내부 등 사유 공간은 촬영하지 않음 |
| 보행로 점유 장면 | 사회적 가치/보행권 문제 설명 | 차량번호와 얼굴 마스킹 필요 |
| 버스 도착 직전 혼잡 | 호출 타이밍 필요성 설명 | 기사/승객 식별 최소화 |

제출본에는 사진 원본을 그대로 넣기보다, 필요한 부분만 자르고 얼굴·차량번호는 마스킹해서 사용합니다.

## 5. CSV 작성 방법

기본 템플릿은 [field-observation-template.csv](/Users/yun-iljun/programming/queue-bus/data/field-observation-template.csv)입니다. 현장조사 후에는 원본 템플릿을 덮어쓰지 말고 `data/field-observation-YYYY-MM-DD.csv`로 복사해 작성합니다.

필수 컬럼:

| 컬럼 | 작성 예 |
| --- | --- |
| `observation_date` | `2026-06-03` |
| `observation_time_kst` | `17:15` |

| `target_station_seq` | `38` |
| `route_name` | `M4137` |
| `direction` | `서울→동탄` |
| `bus_arrival_time_kst` | `17:18` |
| `bus_departure_time_kst` | `17:19` |
| `waiting_count_before_bus_arrival` | `34` |
| `observed_boarded_count` | `7` |
| `observed_alighting_count` | `0` |
| `left_waiting_count` | `27` |

| `estimated_queue_length_meters` | `22` |
| `sidewalk_blocked_yn` | `Y` |

## 6. 제출서 반영 문구

현장조사 전에는 다음처럼 씁니다.

> 공식 GBIS API 수집 결과 M4137 퇴근 시간대 서울 도심 후반 정류장에서 승차 가능성이 시간대와 정류장에 따라 크게 달라지는 패턴이 관측되었다. 다만 GBIS만으로 실제 대기열 길이, 이용자가 줄 대신 기다릴 수 있는 공간, 미탑승 인원은 직접 확인할 수 없으므로, 제출 전 현장 관찰을 통해 대기 인원·탑승 인원·미탑승 인원·보행로 점유 여부·분산 대기 가능성을 보완 검증한다.

현장조사 후에는 다음처럼 바꿉니다.

> 현장 관찰에서도 버스 도착 전 대기 인원과 출발 후 미탑승 인원이 확인되어, GBIS 승차 가능성 지표가 실제 이용자 대기 문제와 연결됨을 확인했다. QueueBus는 이 신호를 이용해 이용자에게 다음 버스 승차 가능성과 호출 시점을 안내하고, 운수사에는 시간대별 대기 수요를 사전에 제공한다.
