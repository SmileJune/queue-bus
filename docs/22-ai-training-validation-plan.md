## 3. SeatFlow AI 학습 설계

### 3-1. 예측 문제 정의

SeatFlow AI는 숫자를 예측하는 회귀 문제입니다.

예측 대상:

```text
목표 정류장 도착 시 예상 잔여좌석
또는
목표 정류장에서 실제 탑승 가능 인원
```

MVP에서는 더 안정적인 `목표 정류장 도착 시 예상 잔여좌석`을 1차 목표로 둡니다. `실제 탑승 가능 인원`은 하차·대기열 데이터가 더 필요하므로 2차 목표로 둡니다.

### 3-2. 학습 데이터 구성

학습 데이터는 차량이 목표 정류장에 도착하기 전 상태를 입력으로 하고, 목표 정류장 통과 시점의 잔여좌석 또는 좌석 변화량을 정답으로 둡니다.

입력 특징:

| 특징 | 설명 | 예 |
| --- | --- | --- |
| `route_id` | 노선 ID | M4137 |
| `target_station_seq` | 목표 정류장 순번 | 9 |
| `current_station_seq` | 현재 차량 순번 | 5 |
| `remaining_stops` | 남은 정류장 수 | 4 |
| `current_remain_seats` | 현재 잔여좌석 | 23 |
| `time_bucket_15m` | 15분 단위 시간대 | 08:45 |
| `day_type` | 평일/휴일/공휴일 | weekday |
| `direction` | 출근/퇴근 방향 | 동탄→서울 |
| `recent_target_avg_seats` | 최근 목표 정류장 평균 잔여좌석 | 9.6 |
| `recent_target_min_seats` | 최근 목표 정류장 최소 잔여좌석 | 0 |
| `recent_zero_seat_rate` | 최근 0석 비율 | 0.42 |
| `vehicle_recent_seat_slope` | 같은 차량의 최근 좌석 감소 추세 | -3.5 |
| `headway_proxy_minutes` | 앞차와의 간격 추정 | 12 |

정답 라벨:

| 라벨 | 설명 |
| --- | --- |
| `target_remain_seats` | 목표 정류장 도착/통과 시 잔여좌석 |
| `seat_delta_to_target` | 현재 시점부터 목표 정류장까지 좌석 감소량 |
| `low_seat_yn` | 목표 정류장 도착 시 10석 이하 여부 |
| `zero_seat_yn` | 목표 정류장 도착 시 0석 여부 |

### 3-3. 모델 후보

처음에는 세 모델을 비교합니다.

| 모델 | 역할 | 장점 | 사용 판단 |
| --- | --- | --- | --- |
| 규칙 기반 baseline | 비교 기준 | 설명 쉽고 과적합 없음 | 반드시 포함 |
| Random Forest 회귀 | 첫 학습 모델 | 작은 데이터에도 안정적 | 기준 학습 모델 |
| LightGBM 회귀 | 성능 후보 | 빠르고 표 데이터에 강함 | MVP 1차 적용 모델 |
| XGBoost 회귀 | 성능 후보 | 예측 성능 좋고 검증 사례 많음 | LightGBM과 비교 |
| 피크/비피크 전용 LightGBM | 세그먼트 전용 후보 | 출근 피크와 비피크 패턴을 분리 학습 | 전역 모델보다 좋을 때만 적용 |
| 위험구간 전용 LightGBM | 세그먼트 전용 후보 | 0석/10석 이하 상황을 별도 보정 | 표본 부족 시 전역 모델 fallback |

모델 선택 기준:

1. 검증 MAE가 낮을 것
2. 0석/10석 이하 위험 예측을 잘할 것
3. 과적합 차이가 작을 것
4. 설명 가능한 중요 특징이 나올 것
5. 예측 속도가 충분히 빠를 것

세그먼트 분리 학습 기준:

- 출근 피크/비피크처럼 표본이 충분한 구간은 전용 모델 후보를 실제로 학습해 전역 모델과 비교합니다.
- 0석, 1~10석 같은 위험구간은 서비스 핵심 구간이지만 현재 표본이 작으므로, 단독 모델을 무조건 적용하지 않습니다.
- 위험구간 전용 모델은 최소 학습 행 수를 넘는 구간만 학습하고, 부족한 구간은 전역 LightGBM으로 fallback합니다.
- 최종 적용은 “세그먼트 전용 모델이 해당 세그먼트와 전체 교차검증에서 모두 개선될 때”로 제한합니다.
- 현재 검증 결과는 [24-seatflow-model-comparison.md](/Users/yun-iljun/programming/queue-bus/docs/24-seatflow-model-comparison.md) 기준 전역 LightGBM이 가장 안정적입니다.

### 3-4. 학습/검증 데이터 분리

무작위로 train/test를 섞으면 안 됩니다. 같은 차량의 연속 스냅샷이 train과 test에 동시에 들어가면 모델이 실제보다 잘 맞는 것처럼 보입니다. 이를 데이터 누수라고 합니다.

권장 분리 방식:

| 분리 방식 | 설명 | 목적 |
| --- | --- | --- |
| 시간 기준 holdout | 최근 1일을 검증 데이터로 두고 이전 4일로 학습 | 미래 예측 상황과 유사 |
| 날짜별 cross-validation | 5일 중 하루씩 돌아가며 검증 | 날짜 편향 확인 |
| 차량 그룹 분리 | 같은 차량 ID가 train/test에 동시에 들어가지 않게 일부 실험 | 차량 기억 효과 방지 |
| 방향별 검증 | 출근/퇴근 방향을 따로 평가 | 방향별 패턴 차이 확인 |
| 정류장별 검증 | 샘플 적은 정류장 성능 별도 확인 | 특정 정류장 과적합 확인 |

MVP 권장:

```text
Train: 2026-05-28 ~ 2026-06-01
Validation: 2026-06-02
추가 검증: 날짜별 leave-one-day-out
```

데이터가 5일뿐이므로 검증 점수는 “초기 지표”로만 표현합니다. PoC에서는 최소 3~4주 데이터를 모아 재평가해야 합니다.

### 3-5. 과적합 방지 설계

과적합은 모델이 실제 패턴을 배운 것이 아니라, 학습 데이터의 우연한 패턴을 외운 상태입니다.

방지 방법:

| 방법 | 설명 |
| --- | --- |
| 시간 기준 검증 | 미래 날짜를 검증으로 두어 실제 운영 상황을 모사 |
| 차량 ID 직접 사용 제한 | 차량번호를 그대로 외우지 않게 하거나 중요도를 점검 |
| 정류장 샘플 하한 | 샘플 30건 미만 정류장은 학습 예측 대신 fallback 사용 |
| 모델 복잡도 제한 | 나무 깊이, leaf 수, tree 수 제한 |
| Early stopping | 검증 오차가 더 좋아지지 않으면 학습 중단 |
| 규칙 baseline 비교 | 복잡한 모델이 단순 공식보다 실제로 나은지 확인 |
| 특징 중요도 점검 | 시간대/잔여좌석/남은 정류장 등 합리적 특징이 중요한지 확인 |
| 검증 오차와 학습 오차 비교 | 학습 오차만 낮고 검증 오차가 높으면 과적합으로 판단 |

모델별 제한값 초안:

| 모델 | 과적합 방지 설정 |
| --- | --- |
| Random Forest | `max_depth` 4~8, `min_samples_leaf` 5~20, tree 100~300 |
| LightGBM | `num_leaves` 8~31, `max_depth` 3~6, `learning_rate` 0.03~0.1, early stopping |
| XGBoost | `max_depth` 3~5, `subsample` 0.7~0.9, `colsample_bytree` 0.7~0.9, early stopping |

### 3-6. 성공 기준

MVP 성공 기준:

| 지표 | 목표 |
| --- | --- |
| 규칙 baseline 대비 MAE | 10% 이상 개선 |
| 목표 정류장 잔여좌석 MAE | 5석 이하 |
| 10석 이하 위험 분류 F1 | 0.70 이상 |
| 0석 위험 recall | 0.70 이상 |
| train/validation MAE 차이 | 2석 이하 또는 30% 이하 |

주의:

- 샘플 5일 기준으로는 지표가 흔들릴 수 있습니다.
- 발표에서는 “초기 데이터 기준 모델 비교 결과”로 표현하고, 상용 수준 정확도라고 과장하지 않습니다.

## 4. Demand Forecast AI 학습 설계

### 4-1. 현재 데이터 판단

실제 대기 인원 데이터는 거의 없습니다. 오늘 실사 데이터는 5대 차량 수준이고, 앱 체크인 이벤트는 아직 없습니다. 따라서 “실제 대기 인원 예측 모델”을 지금 학습시키기에는 부족합니다.

대신 현재 가능한 것은 `좌석 위험 기반 수요 압박 예측`입니다.

예:

```text
명동입구는 최근 5일 0석 신호가 높다.
이 시간대에는 사용자가 다음차 안내를 받을 가능성이 높다.
```

### 4-2. 초기 모델

초기에는 학습 모델보다 통계 기반 점수를 사용합니다.

입력:

- 정류장별 최근 0석 비율
- 10석 이하 비율
- 피크 시간대
- 최근 평균 잔여좌석
- 배차 간격

출력:

- 예상 혼잡 수요 압박 등급
- 다음차 안내 필요 가능성

### 4-3. 학습 모델 전환 조건

학습형 Demand Forecast로 전환하려면 아래 데이터가 필요합니다.

| 데이터 | 최소 기준 | 권장 기준 |
| --- | ---: | ---: |
| 앱 체크인 이벤트 | 1,000건 이상 | 10,000건 이상 |
| 관측일 | 7일 이상 | 28일 이상 |
| 정류장별 피크 샘플 | 100건 이상 | 500건 이상 |
| 미탑승/이월 라벨 | 100건 이상 | 1,000건 이상 |

### 4-4. 검증 방식

- 최근 1주를 검증으로 분리
- 평일/휴일을 따로 평가
- 피크 시간대 MAE를 별도 산정
- 혼잡 등급 정확도 평가

### 4-5. 과적합 방지

- 정류장별 샘플이 적으면 전체 모델 + 정류장 통계 보정 사용
- 특정 날짜 이벤트를 외우지 않도록 날짜 ID 직접 사용 제한
- 날씨/공휴일 변수는 충분한 관측 후 투입

## 5. Boarding Probability AI 학습 설계

### 5-1. 현재 데이터 판단

개인별 탑승 가능성 모델은 현재 학습할 수 없습니다. 이유는 `대기번호`, `내 앞 대기 인원`, `호출 여부`, `실제 탑승 성공/실패`가 앱 이벤트로 아직 쌓이지 않았기 때문입니다.

### 5-2. 필요한 라벨

예측 대상:

```text
y = 이번 차량 탑승 성공 여부
```

필요 이벤트:

| 이벤트 | 라벨 생성 용도 |
| --- | --- |
| `check_in` | 대기번호, 체크인 시각 |
| `call_sent` | 호출 대상 여부 |
| `call_ack` | 호출 확인 여부 |
| `arrived_at_queue` | 실제 대기 위치 도착 여부 |
| `boarded` | 탑승 성공 |
| `left_behind` | 미탑승 |
| `no_show` | 호출 후 미도착 |

### 5-3. 초기 규칙 모델

앱 이벤트가 쌓이기 전에는 아래 규칙으로 시작합니다.

```text
effectivePosition = queuePosition - usersAhead * noShowRate
margin = expectedBoardableCount - effectivePosition
```

### 5-4. 학습 모델 전환 조건

| 데이터 | 최소 기준 | 권장 기준 |
| --- | ---: | ---: |
| 탑승 성공/실패 라벨 | 500건 | 5,000건 |
| 정류장별 라벨 | 100건 | 1,000건 |
| 노쇼 라벨 | 100건 | 1,000건 |
| 관측일 | 7일 | 28일 |

### 5-5. 검증·과적합 방지

- 사용자 ID 기준 분리: 같은 사용자가 train/test에 동시에 많이 섞이지 않도록 점검
- 날짜 기준 검증: 미래 날짜 성능 확인
- 확률 보정: 70%라고 안내한 그룹이 실제 70% 탑승했는지 확인
- 개인정보 최소화: 개인 ID는 해시 처리하고 장기 이동 경로는 쓰지 않음

## 6. Call Optimizer AI 학습 설계

### 6-1. 현재 데이터 판단

호출 최적화는 아직 학습할 수 없습니다. 실제 호출을 보낸 적이 없으므로 호출 응답률, 노쇼율, 과다 호출, 과소 호출 라벨이 없습니다.

### 6-2. 초기 규칙

```text
callCount =
  min(maxCallCount, ceil(expectedBoardableCount / (1 - noShowRate)))

callTimeBeforeArrival =
  averageWalkToQueueSeconds + alignmentBufferSeconds
```

### 6-3. 학습 데이터

필요 라벨:

| 라벨 | 의미 |
| --- | --- |
| 호출 응답률 | 호출 후 사용자가 확인했는지 |
| 대기 위치 도착 시간 | 호출 후 실제 이동 시간 |
| 탑승 성공률 | 호출된 사람 중 실제 탑승한 비율 |
| 과다 호출 | 호출됐지만 탑승하지 못한 사람 수 |
| 과소 호출 | 빈 좌석이 남았는데 호출 인원이 부족한 경우 |

### 6-4. 최적화 방식

처음부터 강화학습을 쓰지 않습니다. PoC에서는 정책 후보를 비교합니다.

정책 예:

| 정책 | 설명 |
| --- | --- |
| 보수 호출 | 예상 탑승 가능 인원만큼만 호출 |
| 노쇼 보정 호출 | 예상 탑승 가능 인원 / (1 - 노쇼율) 호출 |
| 혼잡 보수 호출 | 위험 정류장은 노쇼 보정을 줄임 |
| 예비 호출 | 호출 응답률이 낮으면 추가 호출 |

손실 함수:

```text
loss =
  emptySeatPenalty * 빈 좌석 수
  + crowdingPenalty * 과다 호출 수
  + leftBehindPenalty * 미탑승 수
  + waitingPenalty * 물리적 줄서기 시간
```

### 6-5. 과적합 방지

- 정책 수를 너무 많이 만들지 않음
- 정류장별로 따로 최적화하기 전에 전체 정책 성능 확인
- 평일/휴일/날씨별로 성능이 크게 흔들리는지 확인

## 7. Left-behind Risk AI 학습 설계

### 7-1. 현재 데이터 판단

현장 실사로 미탑승 사례가 확인됐지만 5건뿐입니다. 학습에는 부족합니다. 하지만 운영 리스크 신호로는 매우 중요합니다.

### 7-2. 초기 모델

SeatFlow AI의 예상 탑승 가능 인원과 예상 대기 인원을 비교합니다.

```text
expectedLeftBehind =
  max(expectedWaitingCount - expectedBoardableCount, 0)
```

### 7-3. 학습 모델 전환 조건

| 데이터 | 최소 기준 | 권장 기준 |
| --- | ---: | ---: |
| 미탑승 발생 라벨 | 100건 | 1,000건 |
| 미탑승 미발생 라벨 | 300건 | 3,000건 |
| 정류장별 라벨 | 30건 | 300건 |
| 관측일 | 7일 | 28일 |

### 7-4. 검증 방식

- 미탑승은 드문 이벤트일 수 있으므로 정확도만 보면 안 됨
- Recall 중요: 실제 미탑승을 놓치지 않아야 함
- Precision도 점검: 너무 자주 위험 알림을 보내면 운영자가 무시함
- F1-score로 균형 평가

### 7-5. 과적합 방지

- 미탑승 발생 건수가 적을 때 복잡한 모델 금지
- 처음에는 로지스틱 회귀 또는 작은 Random Forest 사용
- 특정 정류장 하나의 실사 사례만 외우지 않도록 날짜/정류장 분리 검증

## 8. Congestion & Safety AI 학습 설계

### 8-1. 현재 데이터 판단

보행로 점유, 줄 길이, 날씨 위험 라벨이 부족합니다. 현재는 점수 기반 모델이 적합합니다.

### 8-2. 초기 점수 모델

```text
congestionScore =
  waitingCount * 0.5
  + expectedWaitMinutes * 0.3
  + weatherRiskScore * 0.2
  + sidewalkRiskBonus
```

### 8-3. 학습 전환 조건

- 현장 관찰 또는 CCTV/운영자 입력으로 `보행로 점유 Y/N` 라벨 확보
- 날씨 데이터 연동
- 정류장 공간 특성 데이터 확보

### 8-4. 과적합 방지

- 날씨 변수는 계절이 충분히 관측된 뒤 사용
- 특정 정류장의 공간 특성이 전체 모델을 왜곡하지 않도록 정류장별 보정 사용

## 9. Anomaly Detection AI 학습 설계

### 9-1. 현재 데이터 판단

실제 사용자 행동 로그가 없으므로 학습형 이상탐지는 아직 불가능합니다.

### 9-2. 초기 규칙

- 체크인 반경 100m 초과: 체크인 불가
- 짧은 시간 내 비현실적 위치 이동: 재인증
- 반복 노쇼: 경고 또는 호출 우선순위 보정
- 동일 기기 다중 계정 체크인: 운영자 확인

### 9-3. 학습 전환 조건

| 데이터 | 최소 기준 |
| --- | ---: |
| 체크인 이벤트 | 10,000건 |
| 노쇼 이벤트 | 1,000건 |
| 위치 이탈 이벤트 | 1,000건 |
| 정상 사용자 기준 데이터 | 4주 이상 |

### 9-4. 과적합·개인정보 방지

- 개인의 정밀 이동 경로를 장기 저장하지 않음
- 좌표 원본 대신 정류장 반경 안/밖, 거리 범주 등으로 축약
- 모델 판단만으로 제재하지 않고 운영자 확인 단계를 둠

## 10. Operator Insight AI 학습 설계

### 10-1. 현재 전략

운영자 요약은 처음부터 학습 모델이 필요하지 않습니다. 계산된 지표를 템플릿 문장으로 바꾸면 됩니다.

예:

```text
55305 이주택지 서울방향은 최근 2시간 평균 잔여좌석이 9.6석이고
0석 신호가 반복되어 다음차 안내 가능성이 높습니다.
```

### 10-2. LLM 고도화

LLM은 숫자 예측 모델이 아니라 리포트 작성 보조로 사용합니다.

입력:

- 정류장별 위험 지표
- 시간대별 혼잡 변화
- 미탑승 반복 구간
- 권장 조치

출력:

- 일일 운영 리포트
- 운수사 배차 검토 요약
- 지자체 제출용 민원 대응 초안

### 10-3. 과적합·환각 방지

- LLM이 숫자를 새로 만들지 못하게 함
- 모든 숫자는 API 계산 결과만 사용
- 문장 템플릿과 근거 표를 함께 제공
- “추정”, “관측”, “예측”을 구분해 표시
