# QueueBus 실서비스 운영 설계

목적: QueueBus를 발표용 MVP가 아니라 실제 정류장 운영 서비스로 본다는 전제에서, 예측 모델·보정 정책·이벤트 로그·안전장치·재학습 운영 기준을 정리합니다.

## 1. 운영 목표

실서비스의 목표는 단순히 잔여좌석을 맞히는 것이 아닙니다.

| 운영 질문 | 서비스 판단 |
| --- | --- |
| 이번 차량에 몇 명까지 안정적으로 호출할 수 있는가 | 안정 호출 인원 |
| 몇 명은 탈 수도 있지만 실패 가능성이 있는가 | 불확실 인원 |
| 몇 명은 다음 차량을 안내해야 하는가 | 다음차 권장 인원 |
| 어느 정류장·시간대가 반복적으로 실패하는가 | 운영자 위험 알림 |
| 예측이 계속 빗나가는 세그먼트가 있는가 | 세그먼트 보정·재학습 대상 |

따라서 QueueBus의 운영 모델은 `예측값`을 그대로 사용자 안내로 쓰지 않고, `예측 -> 보정 -> 운영 판단 -> 결과 라벨 수집 -> 재학습` 흐름으로 동작해야 합니다.

## 2. 운영 모델 구조

| 계층 | 역할 | 현재 적용 |
| --- | --- | --- |
| SeatFlow 예측 모델 | 목표 정류장 도착 시 잔여좌석 예측 | 전역 LightGBM |
| 세그먼트 보정 | 시간대·피크·위험구간별 과대예측 보정 | 학습 residual 기반 보수 보정 |
| 호출 정책 | 안정 호출/불확실/다음차 권장 분리 | 보수 호출 |
| 운영 알림 | 반복 미탑승·과대 안내·위험 정류장 감지 | 규칙 + 이벤트 라벨 |
| 재학습 | 신규 GBIS·체크인·탑승 결과 반영 | 주기 학습 및 승격 심사 |

현재 모델 비교 결과에서는 전역 LightGBM이 가장 안정적입니다. 피크/비피크 전용 모델과 위험구간 전용 모델도 검증했지만 현재 데이터량에서는 전역 모델보다 낮은 성능을 보였습니다. 따라서 운영 초기에는 전역 LightGBM을 기본 모델로 사용하고, 세그먼트별 보정값으로 안전성을 높입니다.

## 3. 보수 호출 정책

사용자에게 “탑승 가능”을 단정하지 않습니다. 안내는 세 단계로 나눕니다.

| 구분 | 의미 | 사용자 안내 |
| --- | --- | --- |
| 안정 호출 | 보정 후에도 탑승 가능성이 높은 인원 | 기존 대기 위치로 이동 |
| 불확실 | 좌석 변동·하차 여부에 따라 달라질 수 있는 인원 | 대기 유지, 추가 호출 가능 |
| 다음차 권장 | 이번 차량 실패 가능성이 높은 인원 | 다음 차량 안내 |

운영 예시:

```text
LightGBM 예측 잔여좌석: 5석
08시대/출근 피크/위험구간 보정: 2석
서비스 보수 예측: 3석
안정 호출: 3명
불확실: 1~2명
나머지: 다음차 권장
```

이 방식은 평균 MAE를 조금 희생하더라도, 탑승 가능하다고 안내한 사용자가 실제로 못 타는 위험을 줄입니다.

## 4. 이벤트 로그 설계

실서비스에서 가장 중요한 것은 예측 실패를 라벨로 남기는 것입니다.

| 이벤트 | 발생 시점 | 핵심 필드 | 생성 라벨 |
| --- | --- | --- | --- |
| `check_in_created` | 정류장 반경 내 체크인 | user_hash, station_id, route_id, queue_no, gps_accuracy_m | 실제 대기 수요 |
| `check_in_expired` | 위치 이탈·시간 만료 | reason, elapsed_seconds | 이탈/노쇼 후보 |
| `queue_position_updated` | 앞순번 변동 | queue_no, people_ahead | 대기열 상태 |
| `bus_snapshot_matched` | GBIS 차량 위치·좌석 수신 | veh_id, plate_no, station_seq, remain_seat_count, eta_seconds | 예측 입력 |
| `prediction_issued` | 사용자/운영자 안내 직전 | model_version, predicted_seats, corrected_seats, stable_call_count | 예측 라벨 기준 |
| `call_sent` | 호출 발송 | queue_range, call_reason, call_deadline | 호출 대상 |
| `call_ack` | 사용자가 호출 확인 | ack_seconds | 호출 응답률 |
| `arrival_zone_entered` | 호출 후 대기 위치 도착 | distance_m, arrived_at | 호출 이행 |
| `boarding_confirmed` | 탑승 성공 확인 | queue_no, veh_id, confirmation_source | 탑승 성공 |
| `boarding_failed` | 미탑승·승차 실패 | queue_no, reason, observed_waiting_count | 미탑승 라벨 |
| `operator_observation` | 현장 실사·운영자 입력 | waiting, boarded, alighted, left_behind | 검증 ground truth |

개인정보 보호를 위해 사용자 식별자는 해시 처리하고, 정밀 위치는 체크인 판정 후 원본을 장기 저장하지 않습니다. 모델 학습에는 정류장 단위 집계와 이벤트 결과 라벨을 사용합니다.

## 5. 모델 승격 기준

새 모델은 정확도만으로 배포하지 않습니다.

| 기준 | 최소 조건 |
| --- | --- |
| 전체 MAE | 기존 모델보다 개선 |
| 출근 피크 MAE | 기존 모델보다 악화되지 않음 |
| 10석 이하 F1 | 기존 모델보다 악화되지 않음 |
| 0석 Recall | 기존 모델보다 악화되지 않음 |
| 2석 이상 과대예측률 | 기존 모델보다 감소 또는 동일 |
| 안정 호출 실패율 | 운영 기준 이하 |

세그먼트 전용 모델은 해당 세그먼트 성능만 좋아서는 안 됩니다. 전체 성능, 핵심 위험구간, 과대예측률이 모두 통과해야 승격합니다.

## 6. 재학습 주기

| 주기 | 작업 |
| --- | --- |
| 매일 | GBIS 수집 상태, 최신 예측 오차, 미탑승 이벤트 확인 |
| 매주 | LightGBM 재학습, 날짜별 holdout, 세그먼트별 성능 비교 |
| 매월 | 모델 승격 심사, 운영 정책 threshold 조정 |
| 이벤트 발생 시 | 반복 미탑승 정류장, API 장애, 배차 변경 구간 긴급 점검 |

운영 초기에는 모델 자동 배포보다 `학습 자동화 + 수동 승인 배포`가 적절합니다. 교통 안내 서비스는 사용자 신뢰가 중요하므로, 검증되지 않은 모델을 자동 승격하지 않습니다.

## 7. 장애·저신뢰 fallback

| 상황 | fallback |
| --- | --- |
| GBIS 좌석 수신 지연 | 최근 정상값 + 보수 보정, 신뢰도 낮음 표시 |
| 모델 예측 실패 | 규칙 baseline으로 전환 |
| 위치 정확도 낮음 | 체크인 보류 또는 재인증 요청 |
| 미탑승 2대 이상 반복 | 보수 호출 모드, 다음차 안내 강화 |
| 예측 과대오차 반복 | 해당 세그먼트 보정값 상향, 운영자 알림 |

사용자 화면에는 “확정 탑승”이 아니라 “안정 호출”, “불확실”, “다음차 권장”으로 표시합니다.

## 8. 현재 적용 결론

현재 SeatFlow 실험은 실서비스 모델의 출발점으로 의미가 있습니다. 다만 실제 서비스 성능을 확정한 것은 아니므로, 운영 초기 구조는 다음과 같이 잡습니다.

- 전역 LightGBM을 기본 예측 모델로 사용합니다.
- 시간대·출근 피크·잔여좌석 위험구간을 feature로 반영합니다.
- 학습 residual 기반 세그먼트 보정값으로 과대예측을 줄입니다.
- 사용자에게는 안정 호출/불확실/다음차 권장을 분리해 안내합니다.
- 체크인·호출·탑승 성공·미탑승 이벤트를 라벨로 수집합니다.
- 세그먼트 전용 모델은 데이터가 충분해지고 승격 기준을 통과할 때만 적용합니다.
